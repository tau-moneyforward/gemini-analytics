# you asked

個人の GitHub のプライベートリポジトリを、所属する会社のチームメンバーに共有したい。どうする？

---

# gemini response

個人のGitHubアカウントにあるプライベートリポジトリを会社のチームメンバーに共有する場合、\*\*「そのコードの所有権をどうしたいか」**と**「会社のセキュリティポリシー」\*\*によって、推奨される方法が大きく3つに分かれます。

状況に合わせて最適な方法を選んでください。

-----

### 1\. コラボレーターとして招待する（一番手軽）

リポジトリの所有権は個人のまま維持し、特定のメンバーだけに見せたい場合の方法です。

  * **メリット:** 手順が簡単。URLが変わらない。
  * **デメリット:** メンバーが増えると管理が面倒。退職者の削除漏れなどのセキュリティリスクがある。
  * **手順:**
    1.  リポジトリの **[Settings]** タブを開く。
    2.  サイドバーの **[Collaborators]** をクリック。
    3.  **[Add people]** をクリックし、同僚のGitHubユーザー名（またはメールアドレス）を入力して招待を送る。
    4.  相手がメールまたはGitHub通知から招待を承認すれば完了。

> **注意:** 会社の規定によっては、「業務に関連するコードを個人アカウントに置くこと」自体が禁止されている場合があります。

### 2\. リポジトリを会社のOrganizationに譲渡する（推奨）

そのコードが業務成果物であり、今後会社で管理すべきものである場合は、リポジトリごと会社（Organization）へ移譲するのが最も適切です。

  * **メリット:** 会社の管理下に入るため、権限管理やセキュリティが適切に行える。
  * **デメリット:** 個人の所有物ではなくなる。
  * **手順:**
    1.  リポジトリの **[Settings]** タブを開く。
    2.  一番下の **[Danger Zone]** までスクロール。
    3.  **[Transfer ownership]** をクリック。
    4.  移譲先のOrganization名と、確認のためにリポジトリ名を入力して転送。
    <!-- end list -->
      * ※転送するには、あなたがそのOrganizationでリポジトリ作成権限を持っている必要があります。

### 3\. 会社のリポジトリとして複製（コピー）を作る

「個人のリポジトリはそのまま残し、会社の環境にコピーを置きたい」場合です。

  * **メリット:** 個人用と会社用を完全に切り離せる。
  * **デメリット:** 2つのリポジトリができるため、同期するには手動での操作が必要。
  * **手順:**
    1.  会社のOrganization内に、新しい空のリポジトリを作成する。
    2.  ローカルのターミナルで以下のコマンドを実行し、会社のリポジトリへPushする。

<!-- end list -->

```bash
# 現在のリポジトリのディレクトリへ移動
cd my-private-repo

# 会社のリポジトリを新しいリモート先(例: work)として追加
git remote add work https://github.com/COMPANY-ORG/NEW-REPO.git

# コードを会社のリポジトリへプッシュ
git push work main
# (すべてのブランチを送る場合は git push --mirror work)
```

-----

### ⚠️ 重要な確認事項（セキュリティ・コンプライアンス）

実行する前に、必ず以下の点を確認してください。

  * **会社のルール:** 多くの企業では、情報漏洩を防ぐため「個人GitHubアカウントの業務利用」や「個人リポジトリへの業務コードの混入」に厳しいルールを設けています。
  * **知的財産権（IP）:** 個人の趣味で作ったコードを会社に提供する場合、権利が会社に移るのか、個人に残るのかを明確にしておく必要があります。

-----

**どちらのケースに近いですか？**
以下のどれに当てはまるか教えていただければ、より具体的なコマンドや設定手順を案内できます。

  * 「今後このコードは**会社の資産**として開発を続けていく」
  * 「あくまで**個人の趣味コード**だが、参考のために同僚に見せたいだけ」
  * 「副業や業務委託契約などで、**一時的に**共有したい」

# you asked

「副業や業務委託契約などで、一時的に共有したい」が近い。チームに所属する人に一括で権限を共有するには？一人ひとり招待するのは現実的じゃない

---

# gemini response

結論から申し上げますと、**個人のアカウント（Personal Account）のリポジトリには、会社の「Team（チーム）」を直接コラボレーターとして追加することはできません。**

GitHubの仕様上、個人リポジトリに追加できるのは「個人のユーザー（User）」のみだからです。

「一人ひとり招待するのは現実的ではない」という状況で、かつ「副業・業務委託」という文脈であれば、\*\*「会社側のOrganizationにリポジトリのコピー（ミラー）を置き、そこでチーム管理してもらう」\*\*方法が唯一かつ最善の解決策になります。

具体的な手順と、なぜこれが良いのかを解説します。

-----

### 推奨：会社側の環境にソースコードをPushする（ミラーリング）

あなたの手元にあるコードを、会社のGitHub Organization内に作った空のリポジトリにPushしてしまう方法です。

#### この方法のメリット

1.  **権限管理を会社に丸投げできる:**
    リポジトリが会社のOrganization下に入れば、会社側は「〇〇Team」に対して一括で権限を付与できます。あなたがメンバー管理をする必要がなくなります。
2.  **契約上の「納品」として明確:**
    「ここまでの成果物を会社の環境に置きました」という形になるため、業務委託の成果物管理として非常にクリーンです。
3.  **個人の環境は汚れない:**
    あなたの個人リポジトリはそのまま手元に残ります。

#### 手順

**Step 1: 会社側にリポジトリを用意してもらう**
クライアント（会社の担当者）に以下のように依頼します。

> 「私の個人アカウントではチーム単位での権限付与ができない仕様のため、**御社のOrganization内に空のリポジトリを作成し、そこに開発チーム（Team）のアクセス権限を付与してください。** その後、私にも『Write（書き込み）』権限をください」

**Step 2: 新しいリポジトリ先を追加してPushする**
リポジトリが用意されたら、あなたのローカル環境で以下のコマンドを打ちます。

```bash
# プロジェクトのディレクトリに移動
cd my-project

# 会社用のリモート先を "client" という名前で追加
# (URLは会社側から教えられたものを使ってください)
git remote add client https://github.com/COMPANY-ORG/PROJECT-REPO.git

# 会社のリポジトリへコードを送信
git push client main
```

これで、コードは会社のリポジトリに入りました。そのリポジトリには既に会社のチーム全員がアクセスできる状態（Step 1で設定済み）になっています。

-----

### どうしても「個人リポジトリ」のまま共有したい場合

「会社側にリポジトリを作ってもらう権限やフローがない」などの理由で、どうしても個人リポジトリを見せる必要がある場合、**GitHubの標準機能でチームを一括招待する方法はありません。**

代替案としては以下の泥臭い方法になります。

1.  **URLを共有して「申請」してもらう:**
    リポジトリのURLを社内のチャット（SlackやTeams）に貼り、「アクセスしたい人はこのURLを開いてアクセス権限をリクエストしてください」と伝えます。あなたは届いた通知メールから「Approve」ボタンを人数分押すだけになります（招待を送るよりは楽です）。

      * *※これを行うには、リポジトリの設定で権限リクエストを受け付ける設定になっている必要がありますが、プライベートリポジトリでは招待されていないとそもそも404になるため、この手は使えないことが多いです。*

2.  **APIでスクリプトを書く（非推奨）:**
    相手のGitHub IDリスト（CSVなど）をもらい、GitHub APIを使ってループ処理で一括招待を送るスクリプトを書く方法です。しかし、一時的な共有のためにこれを行うのはコストが見合いません。

### 結論としてのNext Step

業務委託契約であれば、成果物の管理責任を会社側に持たせる意味でも、\*\*「会社側にリポジトリを作ってもらい、そこにPushする（Step 1 & 2）」\*\*形での提案をおすすめします。

もし「会社側にリポジトリを作るのを依頼する文面」が必要であれば作成しますので、おっしゃってください。

# you asked

会社の GitHub にリポジトリを増やす前に、個人用のプロジェクトであることが分かるように何らかのタグとかフォルダとかを作ってそこに集約するのはどう？

---

# gemini response

そのお考えは非常に重要です。「公式なプロダクトのコード」と「個人の実験的/一時的なコード」が混ざると、会社側も管理に困るからです。

ただし、GitHubの仕様上、**「Organizationの中にフォルダを作って、そこにリポジトリをまとめる」という機能（階層構造）はありません。** リポジトリはすべてフラットに並んでしまいます。

そのため、ご提案いただいた内容を実現するには、以下の3つのアプローチ（現実的な順）のいずれかを使うのが一般的です。

---

### 方法1：リポジトリ名に「プレフィックス（接頭辞）」をつける（推奨）
「フォルダ」の代わりに「名前」でグルーピングする方法です。最も一般的で、誰が見ても一目で分かります。

会社のリポジトリ作成時に、以下のような名前をつけてもらいます。

* **`sandbox-[あなたの名前]-[プロジェクト名]`**
    * 例: `sandbox-yamada-image-processor`
    * 意味: 「これは実験場（Sandbox）であり、ヤマダ個人のものです」
* **`poc-[プロジェクト名]`**
    * 例: `poc-new-search-engine`
    * 意味: 「これは概念実証（PoC: Proof of Concept）のためのコードです」
* **`vendor-[あなたの名前]-[プロジェクト名]`**
    * 意味: 「外部パートナー（Vendor）からの提供物です」

**メリット:** リポジトリ一覧で検索（フィルター）しやすく、並び順も揃うのでフォルダに入っているかのように見えます。

### 方法2：単一の「実験用リポジトリ」を作り、その中のフォルダで分ける
もしプロジェクトが小規模なスクリプト集などであれば、**リポジトリを1つだけ作り、その中にフォルダを切って管理する**方法もあります。

1.  会社に **`yamada-workspace`** や **`external-contrib-repo`** というリポジトリを1つだけ作ってもらう。
2.  そのリポジトリの中にフォルダを作る。
    * `/project-A/`
    * `/project-B/`

**メリット:** リポジトリが増えすぎないため、会社の管理者の負担が少ない。
**デメリット:** **権限管理が「リポジトリ単位」になるため、**「project-Aは見せたいけど、project-Bは見せたくない」という制御ができなくなります（全員に全部見えます）。

### 方法3：Topic（タグ）をつける
リポジトリの右上に表示される「Topics」を設定します。

1.  リポジトリ作成後、歯車マーク（Settings）の隣にある「About」の編集ボタンを押す。
2.  Topics欄に `personal-project` `experimental` `external-contractor` などのタグを入れる。

**メリット:** 検索で絞り込める。
**デメリット:** 一覧画面（リスト）でパッと見たときに気づかれにくい。方法1（名前）との併用がおすすめ。

---

### おすすめの運用フロー

「フォルダで分けたい」という意図を汲むと、**「方法1（名前の工夫）」が最もトラブルが少なくおすすめ**です。

会社側に依頼する際は、以下のように伝えるとスムーズです。

> 「私の個人プロジェクトであることが一目で分かるように、リポジトリ名の先頭に **`personal-[氏名]-...`** や **`sandbox-...`** と付けて作成していただけますか？ そうすれば、御社の本番コードと混同されるリスクを防げます。」

これなら、相手も「配慮ができる人だ」と安心してリポジトリを作ってくれるはずです。

# you asked

個人のリポジトリから会社のリポジトリに中身をごっそり移すには？

---

# gemini response

個人のリポジトリにある\*\*「全履歴・全ブランチ・全タグ」\*\*を、会社のリポジトリへ「ごっそり」コピーする手順です。

手元のローカル環境にあるGitを使って、新しい送信先（会社のリポジトリ）を追加し、そこに全てを流し込むのが最も確実です。

### 前提

  * **コピー元:** あなたのローカルにあるリポジトリ（最新の状態にしておいてください）
  * **コピー先:** 会社側で作ってもらった「空のリポジトリ」のURL
      * 例: `https://github.com/COMPANY-ORG/sandbox-yamada-project.git`

-----

### 手順（コマンド3発で終わります）

ターミナルを開き、プロジェクトのディレクトリで以下の順に実行してください。

#### 1\. 会社のリポジトリを「送信先」として登録する

名前は何でも良いですが、ここでは分かりやすく `company` という名前で登録します。

```bash
# 会社のリポジトリURLを "company" という名前のリモート先として追加
git remote add company https://github.com/COMPANY-ORG/sandbox-yamada-project.git
```

#### 2\. 全てのブランチと履歴を送信する

`--all` オプションをつけることで、mainブランチだけでなく、developやfeatureなど全てのブランチを送信します。

```bash
git push company --all
```

#### 3\. タグ（リリース情報など）も送信する

`--all` だけではタグが含まれないことがあるため、念のためタグも送ります。

```bash
git push company --tags
```

-----

### これでどうなるか？

  * **コードと履歴:** 全て会社のリポジトリに複製されました。「誰がいつコミットしたか」の履歴もそのまま残ります。
  * **今後の作業:**
      * 自分の個人のリポジトリを更新したい時 → `git push origin main`
      * 会社のリポジトリを更新したい時 → `git push company main`
      * と、送信先を使い分けることができるようになります。

### ⚠️ 注意点：移せないもの

このGitコマンドの方法では、**コード（Gitのデータ）以外は移りません。**
以下のようなGitHub上のデータはコピーされないため、必要であれば手動で転記する必要があります。

  * Issues（イシュー）
  * Pull Requests（プルリクエスト）の会話履歴
  * Wiki
  * GitHub Actionsの設定（シークレットキーなど）

コードさえ共有できれば良い（開発が進められる）という状況であれば、上記の手順で完璧です。

# you asked

登録した company は後で消せる？

---

# gemini response

はい、**いつでも簡単に消せます（解除できます）。**

ここで行う「削除」は、あなたのパソコン（ローカル）に登録した「会社リポジトリへのショートカット（宛先設定）」を削除するだけです。
**手元のコードが消えたり、会社側のリポジトリが消えたりすることは一切ない**ので安心してください。

### 削除（解除）するコマンド

ターミナルで以下のコマンドを実行するだけです。

```bash
git remote remove company
```

### 本当に消えたか確認する方法

以下のコマンドを打って、`company` が表示されなければ削除成功です。

```bash
git remote -v
```

**実行結果の例：**

```text
# 削除前（2つある）
origin  https://github.com/YOUR-NAME/my-repo.git (fetch)
company https://github.com/COMPANY/sandbox.git (fetch)

# 削除後（origin つまり自分の個人用だけに戻る）
origin  https://github.com/YOUR-NAME/my-repo.git (fetch)
```

-----

### おすすめの運用

今回のように「一時的に納品するためだけ」であれば、**Pushが終わった直後にこのコマンドで削除してしまう**のが一番安全です。
そうすれば、うっかり間違えて会社のリポジトリに開発中のコードをPushしてしまう事故（誤送信）を物理的に防げます。

# gemini response

PdM（プロダクトマネージャー）が作成した「実験用・使い捨て・本番投入不可」という意図が明確に伝わる `README.md` のテンプレートを作成しました。

### パターン1：標準的・日本語（推奨）

最も汎用的で、意図が明確に伝わるバージョンです。

````markdown
# [アプリ名/プロジェクト名] - Concept Mock

## ⚠️ 重要：取り扱いについて
このリポジトリは、PdMによる**機能検証・UI/UX実験のためのモックアップ（プロトタイプ）**です。
````
