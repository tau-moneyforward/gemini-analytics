# you asked

URL パラメーターを # 以下に続けて書くのってあり？

---

# gemini response

結論から言うと、**「アリ」ですが、通常のパラメータとは仕組みと用途が全く異なります。**

Webアプリケーション（特にSPA）やセキュリティの文脈ではよく使われますが、**サーバー側には送信されない**という決定的な違いがあります。

プロの視点で「いつ使うのか」「何が違うのか」を整理しました。

---

### 1. 二つの書き方の決定的な違い

#### パターンA：標準的なクエリパラメータ
`https://example.com/search?q=apple#top`
* **場所:** `?` が `#` より**前**にある。
* **挙動:** **サーバーにデータが送られます。**
* **用途:** PHPやRailsなどがリクエストを受け取って、DB検索して結果を返す（サーバーサイド処理）。

#### パターンB：ハッシュ以下のパラメータ（今回の質問）
`https://example.com/#/search?q=apple`
`https://example.com/#access_token=xyz123`
* **場所:** `?` が `#` より**後ろ**にある。
* **挙動:** **サーバーには送られません。ブラウザ（JavaScript）の中だけで処理されます。**
    * サーバーは `https://example.com/` へのアクセスしか検知できません。
* **用途:** React, VueなどのSPA（シングルページアプリケーション）、OAuth認証のトークン受け渡し。

---

### 2. 「#以下」を使う主なケース（アリな場面）

#### ① SPAのクライアントサイドルーティング (Hash Routing)
React Router や Vue Router で `HashMode` を使うと、URLは以下のようになります。
`https://app.com/#/users/123?filter=active`

* **理由:** `#` 以下の変更は**ページのリロード（サーバーへの再リクエスト）を発生させない**ため、画面をチラつかせずにJavascriptだけで画面遷移を表現できるからです。
* *※現在は History API を使った `#` なしのルーティングが主流ですが、簡易的なツールやサーバー設定が変えられない環境では現役です。*

#### ② セキュリティ対策（OAuth / OIDC）
Googleログインなどで、認証が終わった後にアクセストークンを戻す際、ハッシュフラグメントで返す方式があります。
`https://client.com/callback#access_token=secret_token`

* **理由:** `?access_token=...` で送ると、サーバーのアクセスログ（URL履歴）にトークンが丸見えで残ってしまうリスクがあるため、**あえてサーバーに届かない `#` 以下** にトークンを含めることでセキュリティを高めています。

---

### 3. 注意点（デメリット）

もし「普通のWebサイト」でこれをやると、以下の問題が起きます。

1.  **サーバー側で値が取れない**
    * PHPやJavaで `request.getParameter("q")` としても、値は空っぽです。JSで `window.location.hash` を解析して取り出す必要があります。
2.  **SEO（検索順位）に弱い可能性**
    * Googleのクローラーは賢くなっていますが、基本的には `#` 以下はページ内のアンカー（目次移動）として扱われるため、別のページとして認識されないリスクがあります。
3.  **OGP（SNSシェア）が効かない**
    * TwitterやLINEでシェアしたとき、`#` 以下の情報によってサムネイルを変えるといった動的な生成がサーバー側でできません。

### 結論
* **React/VueなどのSPAを作っている場合:** **「アリ」** です（ただしモダンなHistoryモードの方が推奨されがち）。
* **認証トークンを受け取る場合:** **「アリ」** です（セキュリティ上の定石）。
* **普通のWebサイト検索やフォーム送信:** **「ナシ」** です。サーバーにデータが届きません。
