### 1\. 簡易ドキュメント（Markdown版）

**API名**: クラウドマイナンバー Internal API (`v1.0.0`)
**Base URL**: `https://my.moneyforward.com/internal_api/v1`

#### ① 従業員のマイナンバー照会

**`POST`** `/offices/{office_id}/employees/{employee_number}/employee_my_number_inquiry`

クラウド給与/社会保険/年末調整から従業員のマイナンバーを取得します。

**パラメータ (Path)**
| 名前 | 必須 | 説明 |
| :--- | :--- | :--- |
| `office_id` | Yes | 事業所ID (Integer) |
| `employee_number` | Yes | 従業員番号 (String) <br>※referred\_event未指定時は暗号化が必要 |

**リクエストボディ (JSON)**
| 名前 | 必須 | 型 | 説明 |
| :--- | :--- | :--- | :--- |
| `referrer_id` | Yes | Integer | 参照者のoffice\_member ID |
| `referred_event` | No | String | 参照元サービス (`payroll`, `social_insurance`, `tax_adjustment`) |
| `purpose_of_uses` | Yes | Array | 利用目的 (`for_health_insurances` など) |

**レスポンス**

  * **200 OK**: `my_number` (12-13桁), `error_id` (null)
  * **400 Bad Request**: エラー詳細 (`warden_not_found`, `my_number_not_provided` 等)

-----

#### ② 扶養家族のマイナンバー照会

**`POST`** `/offices/{office_id}/employees/{employee_number}/employee_dependent_my_number_inquiry`

扶養家族のマイナンバーを取得します。

**パラメータ (Path)**

  * ①と同様 (`office_id`, `employee_number`)

**リクエストボディ (JSON)**
| 名前 | 必須 | 型 | 説明 |
| :--- | :--- | :--- | :--- |
| `referrer_id` | Yes | Integer | 参照者のoffice\_member ID |
| `referred_event` | No | String | 参照元サービス |
| `purpose_of_uses` | Yes | Array | 利用目的 |
| `employee_dependent`| Yes | Object | 扶養家族情報 (`last_name`, `first_name` 必須) |

**レスポンス**

  * **200 OK**: `my_number` (12桁)
  * **400 Bad Request**: `employee_dependent_not_found` 等

# you asked

マネフォ Admina は IDaaS と言える？

---

# gemini response

結論から言うと、**マネーフォワード Admina は IDaaS ではありません。**

業界の定義としては、**SaaS管理プラットフォーム（SMP: SaaS Management Platform）** というカテゴリに分類されます。

なぜ IDaaS と混同しやすいのか、そして決定的な違いは何かを整理します。

### 1. 決定的な違い：SSO（認証）機能の有無

ここが最大の判断基準です。

* **IDaaS (Okta, Entra IDなど):**
    * **役割:** 「認証（ログイン）」を司る。
    * **機能:** SSO（シングルサインオン）を提供し、ユーザーが**ログインする際の入り口**になります。
    * **例:** ユーザーは「Okta」にログインすることで、SlackやZoomに入れます。
* **Admina (SaaS管理):**
    * **役割:** 「管理・可視化」を司る。
    * **機能:** 認証機能は**持っていません**。Adminaを使ってSlackにログインすることはできません。
    * **例:** Adminaは、「誰がSlackのアカウントを持っているか」「そのアカウントは使われているか（無駄なコストになっていないか）」を管理者が確認・操作するためのツールです。

### 2. なぜ「IDaaSっぽい」と感じるのか？

Admina が IDaaS と混同されやすい理由は、**「アカウント（ID）の作成・削除（プロビジョニング）」ができるから**です。

* **共通点:** 入社時にSlackやGoogleのアカウントを自動発行したり、退職時に削除したりする機能は、IDaaS（上位プラン）にも Admina にも存在します。
* **Adminaの強み:** IDaaSは「SSO連携しているアプリ」しか管理できないことが多いですが、Adminaは「SSOしていないアプリ」や「シャドーIT（会社が把握していないアプリ）」まで、APIや明細データから洗い出して管理できる点にあります。

### 3. IDaaS と Admina の比較表

| 機能 | IDaaS (Okta, Entra ID) | マネーフォワード Admina |
| :--- | :--- | :--- |
| **主なカテゴリ** | Identity Access Management | SaaS Management Platform |
| **認証 (SSO)** | **◎ メイン機能** | × できない |
| **多要素認証 (MFA)** | **◎ メイン機能** | × できない |
| **ID作成・削除** | ○ (SCIM対応アプリ中心) | **◎ (API連携で幅広く対応)** |
| **コスト管理** | △ (基本できない) | **◎ (無駄なライセンス発見)** |
| **シャドーIT検知** | △ (SSOしてないものは見えない) | **◎ (会計・メール連携で発見)** |
| **役割のイメージ** | **「会社の鍵」** | **「資産台帳」** |



### 4. 両者の関係性

この2つは競合するものではなく、**共存・補完する関係**です。

多くの企業では、以下のように使い分けています。

1.  **IDaaS (Okta/Entra ID)** を導入して、セキュリティ（SSO/MFA）を強化する。
2.  **Admina** を導入して、IDaaS 自体のライセンス管理や、IDaaS に対応していない細かいSaaSの棚卸し、コスト削減を行う。
3.  Admina と IDaaS を連携させ、IDaaS 上の情報を Admina に吸い上げて一元管理する。

### まとめ

* **認証（ログイン）させたい**なら → **IDaaS** (Okta, Entra ID, Google Cloud Identity)
* **SaaSの棚卸し・コスト削減・退職者IDの消し忘れ防止をしたい**なら → **Admina** (SaaS管理)

# you asked

歩けむ、とか、走れむ、といった表現は古典にある？

---

# gemini response

結論から言うと、**古典文法として「歩けむ」「走れむ」という形は存在しません（誤りです）。**

もしそのような表現を見かけたとしたら、それは現代語の「可能動詞（歩ける）」に無理やり古典の助動詞「む」をつけた誤用か、あるいは「けむ（過去推量）」の接続を間違えている可能性が高いです。

正しい形と、なぜ間違いなのかを解説します。

### 1. 「〜しよう」「〜だろう」と言いたい場合（助動詞「む」）

「む（ん）」は「推量・意志」を表す助動詞ですが、これは動詞の**未然形（アの段）**につくルールがあります。

古典において「歩く」「走る」は**四段活用**の動詞です。

* **歩く（四段）**
    * 未然形：歩**か**（Aru**ka**）
    * 正しい形：**歩かむ**（Aru**ka**mu）
    * 意味：歩こう、歩くだろう

* **走る（四段）**
    * 未然形：走**ら**（Hashi**ra**）
    * 正しい形：**走らむ**（Hashi**ra**mu）
    * 意味：走ろう、走るだろう

「歩**け**む」「走**れ**む」のように、エの段（已然形や命令形、あるいは現代語の可能形）に「む」がつくことはありません。

---

### 2. 「〜しただろう」「〜したそうだ」と言いたい場合（助動詞「けむ」）

もし語尾の響きが「〜けむ」「〜れむ」に近いものを探しているなら、過去推量の助動詞**「けむ」**を使おうとしている可能性があります。
しかし、「けむ」は動詞の**連用形（イの段）**につくルールがあります。

* **歩く**
    * 連用形：歩**き**（Aru**ki**）
    * 正しい形：**歩きけむ**（Aruki**kemu**）
    * 意味：歩いただろう、歩いたとかいう

* **走る**
    * 連用形：走**り**（Hashi**ri**）
    * 正しい形：**走りけむ**（Hashiri**kemu**）
    * 意味：走っただろう

---

### 3. なぜ「歩けむ」という間違いが生まれるのか？

現代人が古文風の言葉を作ろうとするときに、以下のような混同が起きがちです。

1.  **現代語の「可能形」との混同**
    現代語には「歩ける（Arukeru）」「走れる（Hashireru）」という可能動詞があります。これに「む」をつけて「歩けむ（歩けるだろう）」としたくなりますが、古典には「歩ける」という動詞自体が存在しません（古典で「歩ける」と言いたい場合は「歩くこと能ふ」や「歩かる」などと言います）。

2.  **已然形＋ば（〜すれば）の響き**
    「歩け（ば）」「走れ（ば）」という已然形の響きが強いため、そこにそのまま「む」をつけてしまうケースですが、文法的には接続しません。

### まとめ表

| 動詞 | 言いたいこと | 正しい古典表現 | 誤った表現（存在しない） |
| :--- | :--- | :--- | :--- |
| **歩く** | 歩こう / 歩くだろう | **歩かむ** | 歩けむ |
| | 歩いただろう | **歩きけむ** | 歩けむ |
| **走る** | 走ろう / 走るだろう | **走らむ** | 走れむ |
| | 走っただろう | **走りけむ** | 走れむ |

ですので、創作や短歌などで古典の響きを使いたい場合は、**「歩かむ」「走らむ」**とするのが正解です。

    * 助動詞「む」：**む**（Mu）
    * 完成形：**走られむ**

※古典では、四段活用の動詞には「らる」ではなく「る」がつきます（例：書かれる＝書かる）。その「る」が変化して「れ」になっています。

### 2. どういう意味になる？（重要）
現代語の「走れる（可能）」の感覚で「走られむ（走れるだろう）」として使おうとしているかもしれませんが、**古典での意味は少し違います。**

古典の「る・らる」には4つの意味（受身・尊敬・自発・可能）がありますが、この形の場合は主に以下のどちらかになります。

#### A. 尊敬（〜なさるだろう）
最も自然な使い方はこれです。高貴な人や目上の人に対して使います。
* **歩かれむ**
    * 意味：お歩きになるだろう、歩かれるだろう
* **走られむ**
    * 意味：お走りになるだろう

#### B. 受身（〜されるだろう）
文脈によってはこの意味になります。
* **（道が）歩かれむ**
    * 意味：（この道は）歩かれることになるだろう
* **（人の前を）走られむ**
    * 意味：走られるだろう（走って逃げられる、などのニュアンス）

#### ※「可能（〜できる）」の意味について
「歩けるだろう」「走れるだろう」という**可能**の意味で使いたい場合、平安時代などの古い文法では、**打消の言葉（ず）を伴わない肯定文では「可能」の意味になりにくい**という特徴があります。
（「歩かれず＝歩けない」とは言いますが、「歩かる＝歩ける」とはあまり言いませんでした）

ただ、鎌倉時代以降や文脈によっては「可能」ととれる場合もなくはないですが、基本的には**「尊敬（なさるだろう）」**と受け取られる可能性が高い表現です。

### まとめ

* **「歩けむ」** → 文法的に**間違い**。
* **「歩かれむ」** → 文法的に**正しい**。
    * ただし意味は「歩けるだろう（可能）」よりも**「お歩きになるだろう（尊敬）」**や「歩かれるだろう（受身）」とするのが本格的です。

もし創作などで「（能力的に）歩くことができるだろう」と言いたい場合は、**「歩き得（う）べし」**や**「歩くこと能（あた）はむ」**などがより厳密な古典表現になりますが、雰囲気重視であれば「歩かれむ」でも通じなくはありません。

### 1. 「手に入れられるだろう」【受身・自発】
「（自然と）手に入るだろう」「誰かによって獲得されるだろう」というニュアンスの場合です。これが最も「得られむ」の直訳に近いです。

    * 読み：まさに これを うべし
    * 解説：「当（まさに〜べし）」は「当然〜するはずだ」という強い推量です。「きっと手に入るはずだ」という確信がある場合に使います。

* **可得** （か・とく）

* **必得** （ひっ・とく）

# gemini response

**DITA**（ディータ）とは、**Darwin Information Typing Architecture** の略称で、技術ドキュメント（マニュアルや仕様書）を作成・管理するための、**XMLベースの国際標準規格**（OASIS規格）です。

一言で言うと、**「ドキュメントを『章』や『ページ』で書くのではなく、『情報の部品（トピック）』として書き、それをレゴブロックのように組み合わせてマニュアルを作る仕組み」**のことです。

製造業、ソフトウェア開発、医療機器メーカーなど、大規模で複雑なドキュメントを扱う現場で広く使われています。

---

### 1. DITAの最大の特徴：トピック指向
従来のWordのような執筆方法と決定的に違うのが、**「トピック指向（Topic-based Writing）」**という考え方です。

* **従来:** 第1章から順番に、長い文章を書いていく。
* **DITA:** 情報を**「トピック」**という小さな単位（部品）に切り分けて書く。

#### DITAにおける3つの基本トピック型
DITAでは、書く内容によって「型」が決まっています。

1.  **Concept（概念）**
    * **「それは何か？」**を説明する。
    * 例：「Adminaとは何か」「SSOの概要」
2.  **Task（タスク）**
    * **「どうやるのか？」**（手順）を説明する。
    * 例：「ユーザーを追加する手順」「パスワードをリセットする方法」
3.  **Reference（参照）**
    * **「仕様・データ」**を説明する。
    * 例：エラーコード一覧、パラメータ表、API仕様

### 2. DITAマップ（構成図）
バラバラに書いた「トピック」を、**DITAマップ**という設計図（目次のようなもの）で繋ぎ合わせます。

* **マップA（初心者向けガイド）** = 「概要トピック」＋「基本操作トピック」
* **マップB（管理者マニュアル）** = 「概要トピック」＋「詳細設定トピック」＋「エラーコード表」

このように、**「概要トピック」という一つの部品を、複数のマニュアルで使い回す（再利用する）**ことができます。

---

### 3. DITAを導入するメリット

大規模なドキュメント制作において、以下のような強力なメリットがあります。

* **コンテンツの再利用 (Content Reuse)**
    * 同じ注意書きや手順をコピペする必要がありません。一箇所修正すれば、それを参照している全マニュアルに修正が反映されます。
* **マルチチャネル出版**
    * 一つのDITAソース（XML）から、PDF、HTML、Webヘルプ、EPUBなどを自動生成できます（DITA Open Toolkitというツールを使います）。
* **翻訳コストの削減**
    * 変更があった「トピック（部品）」だけを翻訳に出せば良いため、翻訳料金を大幅に下げられます。
* **構造の強制**
    * XMLタグで「ここは手順」「ここは結果」と厳密に決まっているため、誰が書いても文書構造が統一されます。

### 4. デメリット・課題

* **学習コストが高い:** XMLの知識や、独特のライティングルールを覚える必要があります。
* **導入コストが高い:** 執筆には専用のエディタ（Oxygen XML Editorなど）や、管理システム（CCMS）が必要になることが多く、高額になりがちです。
* **小規模には不向き:** 10ページ程度のメモ書きや、頻繁に構造が変わるブログ記事などには、Markdownなどのほうが向いています。

---

### まとめ

| 項目 | 特徴 |
| :--- | :--- |
| **名前** | Darwin Information Typing Architecture |
| **形式** | XML (構造化データ) |
| **思想** | 部品化して使い回す（トピック指向） |
| **向いているもの** | 製品マニュアル、API仕様書、法律・規約集 |
| **向いていないもの** | 小説、エッセイ、ブログ、短い議事録 |

**「ドキュメントの管理・翻訳コストを下げたい」「マニュアルの表記揺れをなくしたい」**という管理者視点で注目されることが多い技術です。

「列が減った」というのは事実ですが、システム設計上は少し注意が必要です。

* **「徴収額（Input）」の列は消えた**
    給与計算で「いくら天引きするか」を決めるための**保険料額表**においては、区別はなくなり、一般と同じ列を参照すればよくなりました。
* **「属性（Flag）」としての区別は残っている**
    保険料の計算（入り口）は同じになりましたが、**給付（出口）**の計算においては、過去の加入期間の計算や、老齢厚生年金の受給開始年齢の特例などで、「坑内員・船員であった期間」は依然として重要な意味を持ちます。
    また、**船員保険（健康保険部分）**は依然として独自の制度・料率表を持っています。

### まとめ
* **ご推察の通りです。** 制度変更によって料率が統一され、保険料額表から「坑内員」「船員」という**列が消滅（統合）した実績があります。**
* したがって、前回の回答（列は増える可能性がある）と合わせると、**「保険料額表の列構成は、法令改正によって増えることもあれば、減る（統合される）こともある」**と考えるのが最も正確です。

# gemini response

結論から申し上げますと、**「保険料額表（金額または料率）」のテーブルを別途持ち、標準報酬月額マスターとは分けるのが、設計として正しく安全です。**

ただし、**「外部キーとして標準報酬月額マスターの `id` を持つ」のは避けたほうが良い**（または慎重になるべき）です。

その理由と、推奨する設計パターンを解説します。

### 1\. なぜテーブルを分けるべきか？

「標準報酬月額（等級と月額の定義）」と「保険料（いくら払うか）」は、**変化するタイミングと要因（次元）が異なるから**です。

| 項目 | 変化の要因 | 頻度 | 地域差 |
| :--- | :--- | :--- | :--- |
| **標準報酬月額マスター**<br>(等級 = 〇〇円) | 国の制度改正 | 非常に稀<br>(数年に一度あるかないか) | **全国一律** |
| **保険料テーブル**<br>(等級 = 〇〇円徴収) | **都道府県ごとの料率改定**<br>年度の切り替わり | **毎年** (主に3月分から) | **都道府県ごとに違う**<br>(健康保険のみ) |

もし1つのテーブルにまとめてしまうと、「東京都の料率が変わっただけなのに、全国の標準報酬月額データをコピーして更新する」といった無駄が発生し、データの管理が煩雑になります。

### 2\. なぜ `id` で紐付けるのを避けるべきか？

標準報酬月額マスターの `id` (サロゲートキー) に依存すると、運用がつらくなります。

  * **理由:** 保険料テーブルは「**2025年度・東京都・健康保険・20等級**」というビジネスキー（業務的な意味のある値）で決まるべきだからです。
  * **リスク:** もし標準報酬月額マスター側でデータの洗い替え（Delete & Insert）が発生して `id` が変わってしまった場合、保険料テーブルの紐付けがすべて壊れてしまいます。

**推奨:** 外部キー（または論理的な結合キー）には、**`grade`（等級）** を使うのが最も堅牢です。

-----

### 3\. 推奨するテーブル設計案

データの正規化を行い、以下の3つの役割に分ける構成をお勧めします。

#### ① 標準報酬月額マスター (あなたの提示したテーブル)

「等級」と「月額」の定義のみを管理します。これは全国共通の「ものさし」です。

  * ここには「保険料」のカラムは持たせません。
  * ※注意: 健康保険と厚生年金は等級の上限・下限が異なるため、1レコードにまとめる場合は「厚生年金の対象外等級（33級以上）」の扱いに注意が必要です（NULL許容など）。

#### ② 健康保険料テーブル (都道府県別・年度別)

協会けんぽの場合、都道府県ごとに料率（額）が異なります。

```sql
CREATE TABLE health_insurance_premiums (
    id bigint unsigned AUTO_INCREMENT PRIMARY KEY,
    prefecture_code char(2) NOT NULL,       -- 都道府県コード (例: '13' 東京)
    year_month_start int NOT NULL,          -- 適用開始年月 (例: 202403)
    year_month_end int,                     -- 適用終了年月 (NULLなら現在有効)
    
    grade int NOT NULL,                     -- ★標準報酬月額マスターの health_insurance_grade と紐付く
    
    monthly_remuneration int NOT NULL,      -- 月額（マスターの値をコピーして持つことが多い。確認用）
    
    total_amount int NOT NULL,              -- 全額（折半前）
    employee_amount int NOT NULL,           -- 従業員負担額
    employer_amount int NOT NULL,           -- 事業主負担額
    
    care_insurance_type tinyint NOT NULL    -- 0:介護なし(40歳未満), 1:介護あり(40-64歳)
);
```

#### ③ 厚生年金保険料テーブル (全国一律・年度別)

厚生年金は都道府県による差はありませんが、料率が変わる可能性があるため、期間で管理します。

```sql
CREATE TABLE welfare_pension_premiums (
    id bigint unsigned AUTO_INCREMENT PRIMARY KEY,
    year_month_start int NOT NULL,
    year_month_end int,
    
    grade int NOT NULL,                     -- ★標準報酬月額マスターの welfare_annuity_insurance_grade と紐付く
    
    total_amount int NOT NULL,
    employee_amount int NOT NULL,
    employer_amount int
);
```

### 4\. 運用イメージ

**給与計算時のロジック:**

1.  従業員の「標準報酬月額」が決まる（例: 30万円）。
2.  **標準報酬月額マスター** を引いて、等級を取得する（例: 22等級）。
3.  従業員の「事業所（都道府県）」と「計算対象月」と「年齢（介護保険有無）」を確認する。
4.  **保険料テーブル** を検索する。
      * `WHERE prefecture_code = '13' (東京)`
      * `AND year_month_start <= 202504`
      * `AND grade = 22`
5.  ヒットしたレコードの `employee_amount` を控除額として採用する。

### まとめ

  * **別テーブルにする？**: **YES**。絶対に分けたほうがいいです。
  * **IDで紐付ける？**: **NO**。`grade`（等級）で紐付けるのが安全です。
  * **設計のポイント**: 健康保険には「都道府県」という軸が必須であることを忘れないようにしてください（組合健保の場合は別ですが、協会けんぽを想定する場合）。
