# you asked

DB の constraint って言葉はある？

---

# gemini response

はい、**DB（データベース）の用語として「Constraint（コンストライント）」は非常によく使われます。**

日本語では\*\*「制約（せいやく）」\*\*と訳されます。

データベースに保存されるデータが「正しい状態」であることを保つために、\*\*「守らなければならないルール」\*\*をテーブルや列（カラム）に設定することを指します。これがないと、データベース内が矛盾したデータやゴミデータだらけになってしまいます。

代表的な5つのConstraint（制約）を紹介します。

-----

### 主要な 5つの Constraint（制約）

開発現場で必ず出てくるのは以下の5つです。

#### 1\. NOT NULL 制約（非NULL制約）

  * **ルール:** 「空っぽ（NULL）にしてはいけない」
  * **例:** ユーザー登録時の「メールアドレス」や「パスワード」。これらが空だと困るので設定します。

#### 2\. UNIQUE 制約（一意性制約）

  * **ルール:** 「他の行と重複してはいけない」
  * **例:** 「社員番号」や「ログインID」。AさんとBさんが同じIDを持っていたらシステムが壊れるため、重複をエラーにします。

#### 3\. PRIMARY KEY 制約（主キー制約）

  * **ルール:** 「行を特定するための唯一無二のID」
  * **中身:** 実は **「NOT NULL」 ＋ 「UNIQUE」** の最強セットです。
  * **例:** ユーザーID、商品IDなど。テーブルに1つだけ必ず設定します。

#### 4\. FOREIGN KEY 制約（外部キー制約）

  * **ルール:** 「別のテーブルに存在するデータしか入れてはいけない」
  * **例:** 「注文テーブル」の「ユーザーID」列。
      * 「会員テーブル」に存在しない幽霊会員のIDで注文が入ると困ります。この制約があれば、「会員テーブルにないID」を登録しようとするとエラーではじいてくれます。

[Image of relational database foreign key diagram]

#### 5\. CHECK 制約

  * **ルール:** 「特定の条件式を満たさないといけない」
  * **例:** 「年齢」は0以上でなければならない（マイナスの年齢は登録させない）、など。

-----

### 具体的なイメージ（SQL）

テーブルを作るとき（`CREATE TABLE`）、以下のように指定します。

```sql
CREATE TABLE Users (
    idINT PRIMARY KEY,          -- 主キー制約（空ダメ、重複ダメ）
    email VARCHAR(255) UNIQUE,  -- UNIQUE制約（重複ダメ）
    age INT CHECK (age >= 0),   -- CHECK制約（0以上のみ）
    name VARCHAR(50) NOT NULL   -- NOT NULL制約（空ダメ）
);
```

### なぜ Constraint が大事なのか？

もしアプリ（プログラム）側のバグで、ありえないデータ（マイナスの年齢や、重複したIDなど）が送信されてきても、**DB側で「そのデータは受け取れません！」と門前払いして、データを守ってくれるから**です。これを「データの整合性（Data Integrity）を保つ」と言います。
