# you asked

一時テーブルって高速に作成と廃棄ができるもの？

---

# gemini response

結論から言うと、\*\*「通常のテーブルよりはずっと高速だが、タダではない（多少のオーバーヘッドはある）」\*\*です。

「100件〜数千件の検索条件を渡す」という目的において、毎回 `CREATE TABLE` と `DROP TABLE` を繰り返すのが最適かどうかは、**データベースの種類（製品）と頻度**によります。

プロの視点で「本当はどうすべきか」を3つのレベルで解説します。

-----

### 1\. なぜ「高速」と言えるのか（メリット）

一時テーブル（Temporary Table）は、通常の永続テーブルとは扱いが異なります。

  * **メモリ上で処理される:** 多くの場合、ディスク（HDD/SSD）に書き込まず、メモリ上だけで完結するため高速です。
  * **ログが最小限:** データの復旧（リカバリ）が不要なため、重たいログ書き込み（Redo/Undoログ）が省略または簡略化されます。
  * **自動お掃除:** セッション（接続）が切れると勝手に消えるため、ゴミが残る心配がありません。

### 2\. でも「落とし穴」がある（デメリット）

いくら速いと言っても、`CREATE` 文を実行するたびにデータベースの\*\*「管理台帳（システムカタログ）」\*\*への書き込みが発生します。

  * **頻度が高いと遅い:**
    例えば「1秒間に100回実行されるAPI」の中で毎回 `CREATE` と `DROP` を繰り返すと、データベース全体の負荷（カタログロックなど）になり、**逆に遅くなります**。
  * **PostgreSQLの弱点:**
    特にPostgreSQLでは、頻繁な `CREATE TEMP TABLE` はシステムカタログを肥大化させ、「Vacuum（掃除処理）」の負荷を高める原因になるため、**ループ処理内での作成はアンチパターン**です。

-----

### 3\. 【最適解】DB別の「正解」アプローチ

「大量のIDリスト（数千件）を渡して検索したい」場合、一時テーブルを毎回作るよりも良い（または正しい）方法が各DBに用意されています。

#### A. PostgreSQL / MySQL 8.0+ の場合

**「配列（Array）やJSON」で渡すのが最強です。**
わざわざテーブルを作らず、変数を1つ渡すだけで済みます。作成・廃棄のコストはゼロです。

```sql
-- テーブルを作らず、配列をテーブルのように扱う（PostgreSQL）
SELECT * FROM users 
WHERE id = ANY(STRING_TO_ARRAY(?, ',')::int[]); -- ?に "1,2,3...10000" を入れる
```

これなら `WHERE IN` の文字数制限も回避でき、かつ爆速です。

#### B. SQL Server の場合

**「テーブル値パラメータ (TVP)」** を使います。
アプリからデータ構造（ListやDataTable）をそのままプロシージャに渡せます。内部的には一時テーブルに近い動きをしますが、アプリ側で `CREATE` 文を書く必要がなく、最適化されています。

#### C. Oracle の場合

**「Global Temporary Table (GTT)」** を使います。
これは「あらかじめ定義（CREATE）だけしておき、中身はセッションごとに空っぽ」という特殊なテーブルです。

  * **開発時に:** `CREATE GLOBAL TEMPORARY TABLE ...` （1回だけやる）
  * **実行時に:** `INSERT` して使う。
    これなら作成・廃棄のコストはかかりません。

-----

### まとめ：どう実装すべきか？

SQLの `WHERE IN` に入れる件数によって判断を変えます。

1.  **100件〜500件程度:**
      * 気にせず `WHERE IN (...)` でOK。これが一番速いです。
2.  **数千件〜数万件:**
      * **PostgreSQL/MySQL:** 配列やJSON関数を使う（テーブルは作らない）。
      * **SQL Server:** テーブル値パラメータを使う。
      * **どうしても一時テーブルを使うなら:**
          * 「1時間に1回のバッチ処理」なら `CREATE/DROP` しても全く問題ありません。
          * 「ユーザーが頻繁に検索ボタンを押す画面」なら、避けたほうが安全です（DBサーバーのCPUが跳ね上がる可能性があります）。

UIの検索機能などで実装する場合は、**まずは1,000件程度を上限（バリデーション）にして `WHERE IN` で実装し、将来困ったら配列渡し等に変える**のが、開発コストとパフォーマンスのバランスが良い選択です。

# you asked

子ども・子育て支援金制度の正式な英名を調査して

---

# gemini response

こども家庭庁（Children and Families Agency）および関連法の公式英訳に基づくと、以下の表記が正式、または公的文書で最も一般的に使用される表現です。

### 1. 正式名称・推奨表記
**Children and Child-rearing Support Contribution**
（子ども・子育て支援金）

「制度」を含めてシステム全体を指す場合は、以下のように表現します。

* **Children and Child-rearing Support Contribution System**
* **Children and Child-rearing Support Contribution Scheme**

### 2. 解説と根拠
日本の公的機関（こども家庭庁、厚労省、日本法令外国語訳データベース）では、用語を以下のように定義しています。

* **子ども・子育て支援法**: *Act on Children and Child-rearing Support*
* **支援金**: *Support Contribution*

単なる「Fund（基金）」ではなく、社会保険料に上乗せして徴収される「拠出金」の性質を持つため、**"Contribution"** という単語が使われるのがポイントです。

### 3. 文脈別の使い分け（UI・説明文向け）

* **最も正確な表記（公的文書・契約書など）:**
    > **Children and Child-rearing Support Contribution**
    > *※「制度（System）」をつけず、この用語だけで「支援金そのもの」や「徴収の仕組み」を指すことが多いです。*

* **少し説明的にする場合（ニュース・一般向け）:**
    > **Children and Child-rearing Support Fund System**
    > *※一般の英語話者には "Contribution" よりも "Fund" の方が「お金を集める仕組み」として直感的に伝わりやすいため、メディア等ではこちらが使われることもあります。ただし、厳密な法定義からは少しズレます。*

* **給与明細やUIの項目名（スペースが限られる場合）:**
    > **Child Support Contribution**
    > *※ "Children and Child-rearing" は長すぎるため、実務上は "Child Support" と短縮されることが予想されます。*

### まとめ
公的な正確さを期すなら **"Children and Child-rearing Support Contribution (System)"** を使用することをおすすめします。
