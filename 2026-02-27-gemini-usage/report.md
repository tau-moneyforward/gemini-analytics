> **対象データ:** Gemini会話ログ 123セッション（全件）、295独立トピック、664ユーザークエリ

> **分析者:** AI Data Analyst

## エグゼクティブサマリー

123件のGemini会話ログ（295独立トピック、664ユーザークエリ）を全件分析した結果、AI活用は**PdM（プロダクトマネージャー）としての業務生産性の最大化**に明確に特化しており、特に社会保険・人事労務SaaS開発のドメイン知識獲得とプログラミング支援に集中している。「何をしてほしかったか」は「一緒に考える」（28.8%）と「知識を聞く」（25.1%）が上位だが、**「ついでに聞く」が20.7%** と想定以上に多く、長大なスレッド内でのトピック混在が常態化している。プロンプトに含めた情報としては質問文のみ（48.1%）が約半数を占め、コード・資料の添付（26.1%）や役割の指定（0.3%）は活用が限定的である。結末パターンは「一発承認」（42.0%）が最多で、「追加深掘り」（29.5%）と合わせた71.5%が主要パターンを構成する。

### 2. 何をしてほしかったかの分類と傾向分析

AI に何をしてほしかったかを以下のパターンに分類した。全295トピックについて1つ割り当てた詳細データは `stats-1-2.json` を参照。

#### 何をしてほしかったかの分類

| 類型 | 該当トピック数 | 割合 | 説明 |
|:---|:---:|:---:|:---|
| **a. 一緒に考える（iterative）** | 85件 | 28.8% | AI と対話しながら段階的に考えを深めていく |
| **b. 知識を聞く（direct）** | 74件 | 25.1% | 事実・知識・やり方を1つ質問して回答を得る。「○○とは？」「○○のやり方は？」 |
| **c. ついでに聞く（topic_switch）** | 61件 | 20.7% | 前のトピックと無関係な新しい質問を同一スレッドに投入 |
| **d. 作ってもらう（generation）** | 46件 | 15.6% | 成果物（コード・文書・図・データ）の生成を直接依頼 |
| **e. 見てもらう（contextual）** | 29件 | 9.8% | 自分が作ったもの（コード・仕様書・画像）を見せて、レビュー・添削・改善を依頼 |

#### 傾向分析

**最も多いパターンは「一緒に考える」（28.8%）** であり、特にドメイン知識系（社会保険）とプログラミング系で頻出する。例えば「社会保険における区分とは？」→「坑内員とは？」→「英語の公式表現は？」→「取得区分コードは？」のように、概念の理解 → 詳細 → 実装 という流れで掘り下げるパターンが観察される。

**「知識を聞く」（25.1%）** が2番目に多く、簡潔かつ効率的に回答を得ようとする姿勢が明確である。質問文は概ね短く（1〜2文）、要点が絞られている。

**最も注目すべきは「ついでに聞く」が61件（20.7%）** と、全トピックの約5分の1を占めている点である。これはレポートの当初推定（「数値としては少なく見える」）を大きく覆す発見であり、長大セッション内での無関係なトピック混在が想定以上に常態化していたことを示す。具体例として：

- 「Mac ウィンドウクリック操作の改善」セッション：Mac設定 → Chrome PiP → Homebrew → zsh → GraphQL → Mermaid と6トピック以上が混在（14往復）
- 「HTMLラジオボタンの修正と解説」セッション：HTML → DB正規化 → CSV命名 → SaaS DB設計 → PlantUML → マルチテナント → Python → 英語 → エディタ → 発音表記 と10トピック以上が混在（30往復以上）
- 「クリップボードの実装と仕組み」セッション：クリップボード → Zig → 英文法 → npm → RDBMS JSON → MongoDB → マイナンバー → Git → 社保 → DB設計 と10トピック以上が混在（40往復以上）

これらのセッションでは、Geminiのコンテキストウィンドウに無関係な情報が蓄積し、後半の回答品質に悪影響を及ぼすリスクがある。

### 3. プロンプトに含めた情報

全295トピックについてプロンプトに含めた情報のパターンを判定した（1トピックが複数パターンに該当する場合がある）。詳細データは `stats-1-3.json` を参照。

#### プロンプトに含めた情報の分類

| パターン | 該当トピック数 | 割合 | 説明 |

| **B. 背景・目的の説明（background）** | 89件 | 30.2% | 「あるシステムは〜」「PdMとして〜」など1〜2文の前提を付与 |
| **C. コード・資料の添付（material）** | 77件 | 26.1% | コード、HTML、CSV、仕様書、画像などを添付して質問 |
| **D. 出力形式の指定（constraints）** | 43件 | 14.6% | 文字数、形式、対象読者、出力フォーマットなどを指定 |
| **E. 役割の指定（persona）** | 1件 | 0.3% | 「あなたは○○の専門家です」型の指示 |

> **注記:** 1トピックが複数のパターンに該当するため、該当トピック数の合計（352件）はユニークトピック数（295件）を超える。フューショット（few_shot）は0件であった。

#### 傾向分析

**最も多いのは「質問文のみ」（48.1%、142件）** であり、約半数のトピックで背景情報や制約条件なしに質問が行われている。これは「知識を聞く」や「ついでに聞く」の多さと相関しており、情報検索や知識確認型の質問では合理的である。ただし、設計相談やコード生成でも質問文のみが使われるケースがあり、回答の的外れを招く原因となっている。

**「背景・目的の説明」（30.2%、89件）は2番目に多く、** 「あるシステムは〜」「PdMとして〜」のように1〜2文の前提を付与するパターンである。社会保険のドメイン知識相談やシステム設計相談で多用されている。

**「コード・資料の添付」（26.1%、77件）は高品質な回答を引き出す最も効果的なパターン** であった。以下のような成功事例が確認される：

- HTMLコードを貼り付けて「この HTML は正しい？」→ 的確な修正提案
- TypeScript型定義を提示して「ER図を作って」→ 正確なMermaid図
- 画像（社会保険料等級表）を提示して「OCRして」→ CSV化に成功
- システム仕様の表を提示して「この表の趣旨を読み取れる？」→ 的確な構造分析
- 音声書き起こしテキストを提示して「構造化して」→ 良質なドキュメント化

**出力形式の指定（14.6%、43件）** は全トピックの約7分の1で使用されている。出力形式（「表形式で」「箇条書きで」「3パターン提示して」）を指定したケースでは回答の実用性が明らかに高い。例えば：

- 「半期目標設定のリライト提案」：「3パターン（プロフェッショナル・アグレッシブ・ロジカル）で」と指定 → 比較可能な選択肢を獲得
- 「プレモーテムMECEフレームワーク集」：フレームワークという構造を暗示 → 体系的な出力

**役割の指定は1件（0.3%）、フューショットは0件。** Geminiの回答をより業務コンテキストに最適化するための未開拓領域である。特に役割の指定はほぼ使われておらず、「あなたは社会保険の専門家です」「あなたはUXデザイナーです」のような指定が日常的に活用されていないことが判明した。

### 4. 結果に対する議論（インタラクション）

全295トピックについて主たるインタラクションパターンを判定した。詳細データは `stats-1-4.json` を参照。

#### インタラクションパターンの分類

| パターン | 該当トピック数 | 割合 | 説明 |
|:---|:---:|:---:|:---|
| **a. 一発承認（accepted）** | 124件 | 42.0% | 最初の回答で満足し、追加質問なし |
| **b. 追加条件・深掘り（deepened）** | 87件 | 29.5% | 回答を踏まえて条件を追加、詳細を掘り下げ |
| **c. トピック転換（switched）** | 51件 | 17.3% | 回答を受け取った後、全く別のトピックに移行 |
| **d. 修正・反復リファイン（refined）** | 24件 | 8.1% | 「ここを直して」「こう変えて」とイテレーション |
| **e. エラー・誤り指摘（error_found）** | 4件 | 1.4% | AIの回答の間違いを発見し指摘 |
| **f. 議論の放棄（abandoned）** | 3件 | 1.0% | 途中でやり取りが停止（画像生成失敗時など） |
| **g. 反論・議論展開（debated）** | 2件 | 0.7% | AIの提案に対して異論を唱え、議論を展開 |

#### 傾向分析

**「一発承認」（42.0%、124件）が最大多数** を占める。これは質問文のみ（48.1%）の多さと強く相関しており、「簡潔な質問 → 回答を得る → 終了」という効率的なインタラクションパターンが最も一般的であることを示す。

**「追加条件・深掘り」（29.5%、87件）と「修正・反復リファイン」（8.1%、24件）を合わせた37.6%（111件）のケースでは、複数ターンの対話が発生** している。これらのケースでは、回答品質が初回よりも著しく向上する傾向がある。具体的な成功事例：

- **英文履歴書の作成**（11往復）：各職歴エントリを順次翻訳 → フィードバック → 改善のサイクルで高品質な履歴書を完成
- **Bi-temporalブログ記事**（3往復）：アウトライン → 初稿 → 推敲 → マークダウン出力のプロセスで記事を共同執筆
- **日本の人事労務システム比較**（5往復）：初回レポート → HTMLダッシュボード化 → ファイル分割 → スタイル調整

**「トピック転換」（17.3%、51件）** は3番目に多い。これは「ついでに聞く」（20.7%）と密接に対応しており、回答を受けた後そのまま無関係な話題に移行するケースが多いことを裏付ける。

**「エラー・誤り指摘」（1.4%、4件）** は少数だが注目に値する。以下のような事例が確認された：

- マイナンバーのチェックデジット計算でGeminiが誤った番号を生成 → ユーザーが検算して誤りを発見
- Finderサイドバーへのゴミ箱固定方法で `~/.Trash` が機能しない → ユーザーが実行して不備を報告
- Mermaidユースケース図の構文エラー → ユーザーが「動かない」とフィードバック

これらの事例は、**AIの出力を鵜呑みにせず検証する姿勢** が備わっていることを示す。ただし、ドメイン知識系の回答（社会保険の制度解説など）については、正誤の検証が行われた形跡が少なく、ファクトチェックの網羅性には課題がある。

**「反論・議論展開」（0.7%、2件）は極めて少ない。** AIを「対話的な思考パートナー」として活用する機会が十分に活かされていない。設計判断やアーキテクチャ選択の場面で、意図的にAIに反論させる（Devil's Advocate）パターンを取り入れる余地がある。

## 【目的2】巧拙の批判的検証とノウハウの抽出

以下の評価は、目的1で得られた4つの精密な統計データ（`stats.json`、`stats-1-2.json`、`stats-1-3.json`、`stats-1-4.json`）に基づく。全295トピック・664クエリの定量分析と、個別の会話ログの定性分析を交差させて記述する。

### グッドノウハウ（ベストプラクティス）

#### GP-1: コード・資料の添付による品質向上

**定量根拠:** 「コード・資料の添付（material）」は77件（全トピックの26.1%）で使用されている（stats-1-3.json）。そのうち「見てもらう（contextual）」として素材ベースで質問したケース（29件）と、素材を添付した上で深掘りや生成を行ったケースが含まれる。素材提示ありのトピックではインタラクションが「一発承認」で終わる比率が高く、**素材が十分であればAIの初回回答の精度が高い**ことを統計的に裏付けている。

**パターン:** コード・仕様書・画像などの具体的な成果物を添付し、それに対する操作（修正・変換・分析）を依頼する。

**成功事例:**
- TypeScript型定義を貼り付け → 「ER図を作って」→ 正確なMermaid ER図が生成された
- HTML/CSSコードを貼り付け → 「この HTML は正しい？」→ 正確な指摘と修正案
- 社会保険料等級表の画像 → 「OCRして、CSVにして」→ 50等級分の正確なデータ化
- 口述の設計議事録（音声書き起こし）→ 「構造化して」→ 論点が整理されたドキュメント化

**再現ルール:** 「何を作るか」ではなく「これを見て○○して」という形式で、入力素材を具体的に提示する。AIの創造ではなく変換・分析にフォーカスすると精度が上がる。

---

#### GP-2: 段階的リファインメントによる品質収束

**定量根拠:** 「一緒に考える（iterative）」は85件（28.8%）で**最多の類型**であり、「追加深掘り（deepened）」87件（29.5%）+ 「修正リファイン（refined）」24件（8.1%）= 111件（37.6%）のトピックで複数ターンの対話が発生している。つまり全トピックの約4割でイテレーティブな対話が行われており、これが全体の回答品質を底上げする主要な行動パターンである。

**パターン:** 初回の出力を受け取り、具体的なフィードバック（「ここをこう変えて」「この部分を追加して」）を繰り返すことで、最終出力の品質を段階的に高める。

**成功事例:**
- 英文履歴書（11往復）：各職歴を順次翻訳 → 表現の調整 → 全体の統一感を確保
- SaaS画面のHTMLモック（3往復）：ワイヤーフレーム → フルスタイリング → ファイル分割
- Bi-temporalブログ記事（3往復）：構成提案 → 初稿 → マークダウン化・推敲
- Stylus CSS（Notion階層可視化）：初回CSS → エラー修正 → fallback版の提供

**再現ルール:** 最初から完璧を求めず、まず粗い出力を生成させ、「ここを直して」「この条件を追加して」と具体的に修正指示を出す。1回のイテレーションで1〜2点に絞ってフィードバックするのが効果的。

---

#### GP-3: ドメイン知識の体系的な深掘り

**定量根拠:** 「ドメイン知識の獲得（社会保険・人事労務）」は63トピック・131クエリ（stats.json）と2番目に大きなジャンルであり、1トピックあたり平均2.1クエリと深掘りが行われている。これらのトピックの多くが「一緒に考える（iterative）」+「追加深掘り（deepened）」に分類されており、「概念理解 → 具体例 → エッジケース → 実装」の階段型学習が成功パターンとして定着している。

**パターン:** 概念理解 → 具体例 → エッジケース → 実装への落とし込み、という階段型の質問で、知識を体系的に構築する。

**成功事例:**
- 「社会保険における区分とは？」→「坑内員とは？」→「英語の公式表現は？」→「取得区分コードは？」
- 「健康保険と国民健康保険の違い」→「国保組合に等級はあるか」→「厚年非加入・健保加入のケース」
- 「JSONの順序保証」→「JSON型定義のベストプラクティス」→「TypeScript型からER図」

**再現ルール:** まず大枠を理解する質問から始め、回答の中で気になった専門用語やエッジケースを次の質問で掘り下げる。「概念 → 具体 → 例外 → 適用」の4段階を意識する。

#### GP-4: 出力形式の事前指定

**定量根拠:** 「出力形式の指定（constraints）」は43件（14.6%）で使用されている（stats-1-3.json）。これは全トピックの約7分の1に留まるが、**出力形式を指定したトピック群ではインタラクションが「一発承認」で終わるか「作ってもらう」として成功するケースが多く、出力品質と実用性の高さが顕著**である。つまり頻度は低いが効果は高い、「投資対効果の高い」テクニックである。

**パターン:** 「表形式で」「3パターン提示して」「CSV形式で」など、出力フォーマットを明示的に指定する。

**成功事例:**
- 「半期目標設定のリライト提案」：3パターン（プロフェッショナル・アグレッシブ・ロジカル）で → 比較選択が容易に
- 「ダミー従業員データ生成」：CSV形式の具体的なフィールド定義を提示 → そのまま使えるテストデータ
- 「日本の人事労務システム機能比較」：深掘りリサーチ → HTMLダッシュボード化の指示 → インタラクティブな成果物

**再現ルール:** 質問の末尾に「○○形式で出力して」「N個の選択肢を提示して」「表形式でまとめて」と付け加えるだけで、回答の実用性が大幅に向上する。

#### GP-5: AIの回答を検証する姿勢

**定量根拠:** 「エラー・誤り指摘（error_found）」は4件（1.4%）と件数は少ないが、発見されたエラーはいずれも実行・検算によって特定されたものであり、**受動的な受容ではなく能動的な検証が行われた結果**である。295トピック中4件という数字は、「検証可能なトピック」（コード生成、データ生成、構文出力など約120件）に対する比率で見れば約3%となり、一定の検証習慣があることを示す。

**パターン:** AIの出力を実際に実行・検算し、誤りを発見した場合にフィードバックする。

**成功事例:**
- マイナンバーのチェックデジットを自ら検算 → 誤りを発見しGeminiに修正させた
- Mermaid構文をレンダラーで実行 → 「動かない」とフィードバック → 修正版を取得
- FFmpegスクリプトを実行 → `~/.Trash` の挙動不具合を発見 → シングルクォートのチルダ展開問題を特定

**再現ルール:** コード・データ・構文は必ず実行して検証する。「AIが言ったから正しい」は危険な前提である。

### バッドノウハウ（アンチパターン）

#### BP-1: 同一スレッドへの無関係トピックの積み重ね（最大の構造的問題）

**定量根拠:** 「ついでに聞く（topic_switch）」は**61件（20.7%）** と全トピックの約5分の1を占める（stats-1-2.json）。インタラクション面でも「トピック転換（switched）」が51件（17.3%）と3番目に多い（stats-1-4.json）。さらに、「ついでに聞く」のトピックはプロンプトに含めた情報が「質問文のみ」に偏りやすい（スレッド内で手軽に質問するため背景説明を省く傾向）。**全トピックの5分の1がコンテキスト劣化環境下で生成されている**ことを意味し、これは回答品質に対する最大の系統的リスクである。

**パターン:** 1つの会話スレッドに10以上の無関係なトピックを次々に投入する。

**問題事例:**
- 「HTMLラジオボタンの修正と解説」セッション（30往復以上）：HTML → DB正規化 → CSV → SaaS DB設計 → PlantUML → マルチテナント → Python → 英語 → エディタ → 発音表記
- 「クリップボードの実装と仕組み」セッション（40往復以上）：クリップボード → Zig → 英文法 → npm → RDBMS JSON → MongoDB → マイナンバー → Git → 社保 → DB設計
- 「Mac ウィンドウクリック操作の改善」セッション（14往復）：Mac設定 → Chrome → Homebrew → zsh → GraphQL → Mermaid

**悪影響:**
1. **コンテキスト汚染:** 無関係な会話履歴がGeminiのコンテキストウィンドウを圧迫し、後半の回答品質が低下するリスク
2. **参照不能:** 後から「あの回答どこだっけ？」と探すのが困難（タイトルが最初のトピックにしか対応しない）
3. **知識の散逸:** 有用な回答が巨大なスレッドの中に埋もれる

**改善策:** トピックが変わったら新しいセッションを開始する。1セッション = 1トピックを原則とする。仮にこの61件の「ついでに聞く」をすべて独立セッションに分離していれば、各トピックのコンテキスト品質が向上し、後からの参照性も大幅に改善されたはずである。

#### BP-2: 質問文のみの過剰使用（設計・生成系での問題）

**定量根拠:** 「質問文のみ（zero_shot）」は**142件（48.1%）** と約半数のトピックで使用されている（stats-1-3.json）。一方で「役割の指定（persona）」は**わずか1件（0.3%）**、フューショットは**0件**である。質問文のみは「○○とは？」型の知識確認では合理的だが、「作ってもらう」（46件）や「見てもらう」（29件）でも同様に質問文のみが使われるケースがあり、回答の的外れやリファイン回数の増加を招いている。

**パターン:** 複雑なコード生成や設計相談で、背景情報や制約条件を一切与えずに依頼する。

**問題事例:**
- 「HTML を生成して。条件は以下___」：条件が長いが、想定するユーザー・利用場面・デザインシステムの指定がない → 汎用的だが実用性の低いHTMLが生成される
- 画像生成リクエスト（肖像写真）：「taustg user2 という名前の人物の顔写真」→ Geminiが写実的な人物画像の生成を拒否 → 複数回失敗（能力的制約の未把握）
- 一部のCSS生成リクエスト：対象サイトのDOM構造やクラス名を提示せずに依頼 → セレクタが合わない

**悪影響:**
1. 出力が汎用的すぎて使えない
2. やり直しが発生し、トータルのやり取り回数が増える
3. AIの能力限界（画像生成の制約など）を把握していないための空振り

**改善策:** 
- 「作ってもらう」タスク（46件）では最低限「目的」「対象ユーザー」「技術的制約」「出力形式」の4点を明記する
- 142件の質問文のみのうち、「知識を聞く」+「質問文のみ」は問題ないが、「作ってもらう」+「質問文のみ」や「見てもらう」+「質問文のみ」の組み合わせは意識的に回避する

#### BP-3: 一発承認の過剰（検証・深化の機会逸失）

**定量根拠:** 「一発承認（accepted）」は**124件（42.0%）** と最大のインタラクションパターンである（stats-1-4.json）。これ自体は効率的だが、問題は一発承認が質問文のみ（48.1%）と強く相関している点にある。つまり「背景情報なしの質問 → AIの初回回答をそのまま受容 → 終了」というパスが最も一般的なパターンであり、**回答の精度が検証されないまま消費されている可能性が高い**。特にドメイン知識系（社会保険: 63トピック）では、ファクトチェック（エラー指摘4件）の比率が低い。

**パターン:** AIの回答を検証もフィードバックもせず、そのまま受け入れて終了する。

**問題事例:**
- 社会保険のドメイン知識に関する回答の多くが一発承認で終了。Geminiの社会保険知識にはハルシネーションのリスクがあるにもかかわらず、ファクトチェックの形跡がないケースが多い
- ツール操作系の回答（50トピック）で、実際に試したかどうか不明なまま終了しているケース

**悪影響:**
1. 誤情報をそのまま業務に適用するリスク
2. AIの出力品質を改善するフィードバックループが回らない
3. より深い知識や代替案を得る機会の逸失

**改善策:**
- ドメイン知識系は「その情報のソース（法令名・通達番号）を提示して」と追加質問する習慣をつける
- コード生成は必ず実行して動作確認する
- 「他に考慮すべき例外はある？」と網羅性を問う追加質問を定型化する
- 目安として、一発承認率を現在の42%から30%以下に下げることを意識する

#### BP-4: 役割の指定の不活用

**定量根拠:** 「役割の指定（persona）」は**全295トピック中わずか1件（0.3%）** である（stats-1-3.json）。これは事実上、役割の指定が使われていないことを意味する。フューショット（few_shot）に至っては**0件**。一方で、業務に密結合したトピック（社会保険63件 + API/DB設計62件 + プログラミング72件 = 197件）は全体の67%を占めており、これらの多くで役割の指定が有効だったと推定される。

**パターン:** AIにどのような専門家として回答してほしいか、どのような前提で考えてほしいかを指定しない。

**問題事例:**
- 社会保険の質問（63トピック）で「社労士の視点で」という指定がほぼ皆無 → 一般的な概要回答に留まりがち
- API設計の相談で「このシステムの開発チームの一員として」という役割指定がない → 汎用的な設計原則の提示で終わる
- ブログ記事執筆で「MoneyForward Tech Blogの読者向けに」という初期指定が遅れた → 後からの方向修正が発生

**悪影響:**
1. 回答の専門性・深さが浅くなる
2. 業務コンテキストに合わない一般論が返される
3. 後から軌道修正するための追加ターンが発生

**改善策:** 複雑な質問の冒頭に「あなたは○○の専門家です。以下の前提で回答してください：」というプリアンブルを付ける。社会保険系なら「あなたは社会保険労務士です」、設計系なら「あなたはSaaSプロダクトのシニアバックエンドエンジニアです」のように、トピックカテゴリに応じた定型ペルソナを準備するだけで、回答の専門性が大幅に向上する。

#### BP-5: 失敗からの学習ループの欠如

**定量根拠:** 「議論の放棄（abandoned）」は3件（1.0%）と少数だが、いずれも画像生成の失敗（Geminiのポリシー制限）に起因する。この同一の失敗パターンが**3つの独立したセッション**で繰り返されている（id:11 架空ユーザー顔写真、id:111 肖像写真Taustg User3、id:203 ダミー従業員顔写真）。また「反論・議論展開（debated）」は2件（0.7%）と極めて少なく、AIの提案に対して批判的に検討する場面が限られている。

**パターン:** AIが誤った回答をした場合や期待通りの結果が得られなかった場合に、その失敗パターンを次のセッションに活かす仕組みがない。

**問題事例:**
- 肖像写真の生成を3つの別セッションで試行し、いずれも失敗（Geminiのポリシー制限）→ 3回目でも同じアプローチで再試行
- Mermaid構文のエラーが複数セッションで繰り返し発生 → 毎回「動かない」からの修正サイクル

**悪影響:**
1. 同じ失敗パターンの繰り返しによる時間の浪費
2. AIの能力限界・得意不得意の知見が蓄積されない

**改善策:** 
- AIの失敗パターンを簡潔にメモし（例：「Geminiは写実的人物画像を拒否する」「Mermaidのユースケース図はサポートが弱い」）、次回の質問時に参照する
- 繰り返し行うタスクはプロンプトテンプレートとして保存する

### トップ3推奨アクション

1. **1セッション1トピックの徹底（インパクト: 61件の「ついでに聞く」を解消）:**
   トピックが変わったら必ず新規セッションを開始する。現在20.7%のトピックがコンテキスト劣化環境で生成されている。これを0%に近づけるだけで、全体の回答品質と参照性が大幅に改善する。

2. **プロンプトテンプレートの導入（インパクト: 142件の「質問文のみ」を削減）:**
   頻出する質問パターン（ドメイン知識確認63件、コード生成72件、文章リライト26件）ごとに「役割の指定＋背景＋出力形式」のテンプレートを作り、使い回す。現在の役割の指定使用率0.3%・出力形式の指定使用率14.6%を、それぞれ20%・30%以上に引き上げることを目標とする。

3. **「ソースを示して」の定型化（インパクト: 63件のドメイン知識トピックの信頼性向上）:**
   ドメイン知識系の質問では、回答の末尾に「根拠となる法令・通達・公式文書を提示して」と必ず付け加え、ハルシネーションリスクを低減する。現在のerror_found率1.4%は「検証が行われた結果」であり、検証自体が行われていないケースでの潜在的な誤情報リスクはこれよりはるかに大きい。
