> スピーカーノート:

- 3 つ例を挙げます

## Slide 4: 成功① 設計意図を伝えるアーティファクト（50秒）

### ① エンジニアに「レビューして」と渡せる仕様書を自分で書く

**入力：** 事業所・社保・労保を網羅した 200 行超の Mermaid ER 図

```
erDiagram
    workplaces { int id PK; string name; ... }
    SOCIAL_INSURANCE_COVERED_WORKPLACES { ... }
    HEALTH_INSURANCE_SETTINGS { ... }
    HEALTH_INSURANCE_RATE_HISTORIES::BI-TEMPORAL { ... }
    PENSION_INSURANCE_SETTINGS { ... }
    LABOR_INSURANCE_COVERED_WORKPLACES { ... }
    ROSAI_INSURANCE_SETTINGS { ... }
    KOYO_INSURANCE_SETTINGS { ... }
```

「これらの ER 図を前提に、GET の API を設計して OpenAPI Spec を書いて」

⬇️

**出力：** Bi-temporal 対応・リソース指向の完全な OpenAPI YAML

```yaml
paths:
  /workplaces:
    get: ...
  /workplaces/{workplaceId}/insurance-histories:
    get:
      parameters:
        - name: target_date      # Bi-temporal 対応
          in: query              # 基準日で時点データを取得
  /social-insurance-workplaces/{id}: ...
  /labor-insurance-workplaces/{id}: ...
components:
  schemas:
    WorkplaceSummary: ...
    InsuranceHistory: ...
    HealthInsuranceSettings: ...
```

> スピーカーノート:
> 「1つ目。自分が設計した ER 図、テーブル10個以上、Mermaidで200行超のものを丸ごと貼り付けて、"これを前提にAPIを設計してOpenAPI Specを書いて"と頼みました。」
> 「返ってきたのが、リソース指向のパス設計、Bi-temporalデータの基準日クエリ、全エンティティのスキーマ定義を含む完全なOpenAPI YAML。」
> 「これまでは設計意図をエンジニアに"口で説明"していた。でもこれがあれば、"このドラフトをレビューして"と渡せる。議論の出発点が変わります。」

## Slide 5: 成功② ドメイン知識の探索（45秒）

### ② 専門家の時間を奪わず、制度のエッジケースまで自分で掘る

例：社会保険の制度理解

```
「社会保険における区分とは？」
  → 「坑内員とは？」
    → 「英語の公式表現は？」
      → 「取得区分コードは？」

「健保と国保の違いは？」
  → 「国保組合に等級はあるか？」
    → 「厚年非加入で健保加入のケースは？」
```

- 概念理解 → 具体例 → エッジケース → プロダクト仕様への落とし込み
- 「ここに例外があるかもしれない」と気づく探索プロセスが仕様策定に直結する

> スピーカーノート:
> 「2つ目。社会保険の制度理解です。人事労務SaaSを作るには、制度のエッジケースまで把握しないと正しい仕様が書けない。」
> 「従来はこれ、社労士や制度に詳しいメンバーに聞くか、厚労省の通達を自分で読むしかなかった。前者は相手の時間を奪う、後者は時間がかかる。」
> 「AIに階段型に質問していくと、概念→具体例→例外→仕様判断を、自席で、自分のペースで回せる。"健保と厚年で等級がずれるケースある？"→"ある"→"じゃあ画面でどう扱う？"を、誰の時間も奪わずにできるようになった。」

---

## Slide 6: 成功③ プロトタイプで即決（35秒）

### ③ その場でデモして、その場で意思決定

- HTML/CSS のプロトタイプを AI で生成
- ステークホルダーとの会議でデモ
- **その場で意思決定まで進んだ**

Before: 「あとでモック作ります」→ 次の会議で共有 → フィードバック → 修正 → ...

After: 「これを見てください」→ 議論 → 決定

> スピーカーノート:
> 「3つ目。AIでHTML/CSSのプロトタイプを作って、それをステークホルダーとの会議でそのままデモしました。結果、その場で意思決定まで進んだ。」
> 「"あとでモック作ります"がなくなった。提案→議論→決定のサイクルが、1回の会議で完結する。PdMとしてこれはかなり大きいです。」

---

## Slide 7: 成功パターン — まとめ（30秒）

### なぜこれらが上手くいったか

## 「構造を渡して、別の構造に変換させる」

- ER 図 → API 仕様書
- 制度の全体理解 → エッジケースの探索
- 設計意図 → 動くプロトタイプ

自分が持っている**構造化された情報**を入力として渡すと、AI の出力精度が格段に上がる

> スピーカーノート:
> 「この3つに共通して、自分が意識していたことがあります。"構造を渡して、別の構造に変換させる"ということです。」
> 「ER図→API仕様書。制度の体系的な理解→エッジケースの探索。設計意図→動くプロトタイプ。いずれも、自分がすでに持っている構造化された情報を入力として渡している。AIにゼロから作らせるのではなく、変換させる。これが精度の鍵でした。」

## Slide 9: 失敗 — ナレッジの散逸（50秒）

### 守備範囲が広がった分、蓄積が失われる

こういう使い方をしていた（全部同じ1つのスレッド）：

1. HTML ラジオボタンの書き方を聞く
2. → DB の正規化の用語を確認する
3. → マルチテナント SaaS の DB 設計を相談する
4. → PlantUML で ER 図を描かせる
5. → 「申請」の英訳を聞く
6. → 法人番号制度について聞く
7. → 顔写真を生成させようとする（失敗）
8. → ...

**往復 56 回。スレッドタイトルは「HTML ラジオボタンの修正と解説」。**

→ マルチテナント DB 設計の重要な議論 10 往復分を、後から探して**見つけられなかった。結構困った。**

> スピーカーノート:
> 「実際のスレッドです。最初はHTMLのラジオボタンについて聞いた。そこからDB設計、マルチテナントの相談、PlantUML、英訳、法人番号制度…全部同じスレッド。56往復。タイトルは"HTMLラジオボタンの修正と解説"。」
> 「この中に、マルチテナントDB設計の重要な議論が10往復分入っていたんですが、後から探そうとして見つけられなかった。結構困りました。」
> 「AIの手軽さが仇になっている。新しいスレッドを開くより、今のスレッドにそのまま打ち込む方が楽。でもその結果、せっかくAIで守備範囲を広げて得た知識が、構造化されないまま消える。改善策はシンプルで、1セッション1トピック。話題が変わったらスレッドを分ける。それだけです。」

## Slide 10: 種明かし①（40秒）

### 実はこれ、全部データで裏付けています

| 指標 | 数値 |
|:---|:---|
| 分析対象セッション数 | **123** |
| 独立トピック数 | **295** |
| 総クエリ数 | **664** |

主な発見：

- 「素材を渡して変換させる」パターン → 全トピックの **26%** で使用、一発で使える出力が返りやすい
- 「1スレッドに複数トピック」 → 全トピックの **21%**。5 回に 1 回
- ゼロショット（背景情報なし） → **48%**
- ペルソナ指定 → **0.3%**（295 トピック中 1 回）

> スピーカーノート:
> 「ここから種明かしです。体感で語ってきましたが、実はGeminiとの全会話ログを定量分析しています。123セッション、独立トピックに分解すると295件、クエリ数は664件。」
> 「"素材を渡して変換"のパターンは全体の26%で、一発で使える回答が返る率が高い。"トピック混在"は21%、5回に1回。体感より多かった。他にも、約半数がゼロショット、ペルソナ指定は295回中1回。伸びしろだらけです。」

## Slide 11: 補足 — 一発承認の読み方（20秒）

### 補足：「一発承認 42%」の解釈

- 会話ログ上は42%が「一発承認」に見える
- しかし、実際には別途 Web 検索でダブルチェックしているケースが相当数含まれる
- **AI との対話に検証プロセスが統合されていないだけで、検証自体はしている**
- 統計はあくまで「AIとの対話ログ」の分析であり、ユーザーの行動全体を映してはいない

> スピーカーノート:
> 「補足です。分析すると"一発承認が42%"と出ました。一見、AIの回答を鵜呑みにしているように見える。でも実際には、Web検索で裏取りした上で承認しているケースがかなり含まれています。つまりAIとの対話に検証のプロセスが残っていないだけで、検証自体はしている。統計の読み方には注意が必要です。」

## Slide 12: 種明かし②（35秒）

### この定量分析自体、AI にやらせました

- Cursor の AI エージェント（Claude）に全 123 ファイルを読み込ませた
- 295 トピックの特定・分類・集計・レポート作成を全自動で実行

```

stats-1-2.json  → 質問手法を 5 パターンに分類
stats-1-3.json  → コンテキストの与え方を 5 パターンに分類
stats-1-4.json  → インタラクションを 7 パターンに分類
report.md       → 上記を統合した 373 行の分析レポート
```

### AI の使い方を分析するために AI を使った

> スピーカーノート:
> 「そしてもう一つ。この定量分析自体、AIにやらせました。Cursor の AI エージェントに123ファイルのGemini会話ログを全部読ませて、トピック特定、4軸の分類、集計、レポート作成を全自動で。出力が4つのJSONと370行のレポート。」
> 「これもまさに"構造を渡して変換させる"パターンです。会話ログという構造化データを渡して、分析レポートに変換させた。」

## Slide 13: まとめ（40秒）

### AI は「不可能→可能」ではなく「やらない→やる」の閾値を動かす

ER 図から API 仕様を書くことも、社保制度のエッジケースを全部調べることも、123 セッションの全件分析も、理論上は手作業で可能。
でも実際にはやらなかった。AI がコストを下げたことで**「やってみよう」**に変わった。

### 明日から使える 2 つのこと

**① 構造を渡して変換させる**

自分が持っている構造化された情報を AI の入力にする。出力の質は入力の質で決まる。

**② 1 セッション 1 トピック**

話題が変わったらスレッドを分ける。守備範囲を広げた分の蓄積を、失わないために。

---

> 「まとめです。AIは魔法ではなくて、できなかったことができるようになるわけじゃない。でも、"面倒だからやらなかったこと"を"やってみよう"に変えてくれる。ER図からAPI仕様を書くのも、制度のエッジケースを全部調べるのも、123セッションを全件分析するのも、全部そう。その積み重ねで、PdMの守備範囲が広がった。」
> 「明日から使える2つのこと。1つ目、構造を渡して変換させる。出力の質は入力の質で決まる。2つ目、1セッション1トピック。広がった守備範囲の分、得た知識を失わないために。」
