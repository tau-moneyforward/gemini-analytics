# Gemini 利用分析レポート v3

## エグゼクティブサマリー

PdM として半年間の Gemini 利用ログ（123 セッション・295 トピック・664 クエリ）を、「与えた入力」×「期待する出力」の 2 軸で分類した。加えて、スレッド衛生（1 セッションに何トピック混在しているか）を独立した指標として記録した。

主な発見：

- **共創（29%）と探索（21%）で全体の半数** — AI を辞書以上の存在として使えている場面が半数ある
- **検索（45%）は最頻出だが価値がないわけではない** — 1-2 回のやりとりで事実を確認し、業務を前に進めている
- **試行（5%）は少数だが示唆的** — 入力なしで成果物を求めると、顔写真生成の失敗のように成功率がばらつく
- **15% のスレッドに 60% のトピックが集中** — 知識が再利用不能になる構造的リスクがある

---

## 1. 分析の枠組み

### 1.1 軸1: 与えた入力（`input_richness`）

ユーザーが AI に渡したプロンプトの情報量を 3 段階で分類する。

#### 素材あり（`material`）: 75 件

ユーザーが**構造化された情報**をプロンプトに貼り付けた、または添付した場合。以下が「素材」に該当する：

- **ソースコード**: TypeScript 型定義、CSS、HTML、シェルスクリプト、正規表現
- **設計図**: Mermaid ER 図、PlantUML 状態遷移図、JSON スキーマ定義
- **仕様書・文書**: Rails スキーマ出力、OpenAPI YAML、CSV 仕様書、法令条文
- **データ**: CSV データ、JSON データ、スプレッドシートの関数式
- **画像**: ワイヤーフレーム画像、社会保険料等級表の画像（OCR 用）、写真（色補正用）
- **テキスト成果物**: 英文履歴書、UI テキスト、ビジネス文書、ブログ下書き
- **エラーログ・ターミナル出力**: コマンド実行結果、エラーメッセージ
- **外部 URL**: 該当なし。URL の提示だけでは素材とみなさず、「文脈あり」に分類した（例: Froala のURL を貼って「これ何？」→ 文脈あり）

「素材あり」の核心は、**AI がそれを直接処理・変換・分析できる構造化された入力を渡している**こと。口頭で「ER 図を作った」と言及するだけでは該当しない。

#### 文脈あり（`contextualized`）: 78 件

素材は渡していないが、**質問の背景や出力の制約を言葉で伝えた**場合。以下が該当する：

- **背景説明**: 「人事労務 SaaS の DB を設計しているのだが…」「社保の手続きで確認したいのだが…」
- **出力形式の指定**: 「表形式で」「3 パターンで」「マークダウンで」「箇条書きで」
- **対象の限定**: 「CSV エクスポート機能のチェックボックスのラベルを提案して」
- **URL の提示**: 外部サイトの URL を示して「これについて教えて」
- **ドメイン文脈の暗黙的な提示**: 具体的な制度名・フォームの項目名への言及

「文脈あり」と「質問だけ」の境界は、**質問文に「なぜそれを聞いているのか」の手がかりがあるかどうか**。「社保の端数処理ルールは？」は社保という文脈がある。「マウスホイールのスクロールが速すぎる」は文脈がない。

#### 質問だけ（`bare`）: 142 件

質問文のみ。背景も素材も形式指定もない。

- 「Statsig ってどう発音する？」
- 「Mac でシェルスクリプトを実行するには？」
- 「malloc ってなんだっけ？」
- 「PlantUML で DB の ER 図って書ける？」

全体の 48% がここに集中する。AI を検索エンジンや辞書の代替として使っているケース。

### 1.2 軸2: 期待する出力（`expected_output`）

ユーザーが AI から得ることを期待した成果の性質を 3 段階で分類する。

#### 成果物（`artifact`）: 72 件

**そのまま使える具体的な成果物**を期待した場合。

- **コード**: HTML/CSS、Chrome 拡張、シェルスクリプト、TypeScript Enum 定義
- **仕様書**: OpenAPI YAML、Mermaid ER 図、PlantUML 図
- **文書**: 英訳、リライト、契約書テンプレート、ブログ記事
- **データ**: ダミー CSV データ、マイナンバーサンプル、UUID
- **画像**: 顔写真生成、写真の色補正、アイキャッチ画像
- **変換結果**: OCR（画像→テキスト）、漢数字→算用数字変換

判定基準: AI の出力をコピーしてそのまま（あるいは軽微な修正で）成果物として利用できるか。

#### 理解（`understanding`）: 90 件

**対話を通じて知識を深める、設計を検討する、レビューを受ける**ことを期待した場合。

- **制度・ドメインの深掘り**: 社保の等級不一致ケース、坑内員の歴史的区分、bi-temporal データの概念
- **設計の壁打ち**: API 設計、JSON 構造、DB 設計、フィールド命名
- **トラブルシュート**: ターミナル表示問題、yarn install エラー、Git 認証
- **レビュー・添削**: HTML の正誤確認、英語表現の添削、JSON 構文チェック
- **比較・調査**: 競合サービスの構造調査、ツール比較

判定基準: 1 回の応答では完結せず、**やりとりを重ねることで理解や判断が深まった**か。あるいは、AI の応答を読んで**自分の知識や設計に反映した**か。単に「合ってる？」と聞いて「合ってます」で終わる場合は「事実」に分類。設計の妥当性を議論した場合は「理解」に分類。

#### 事実（`fact`）: 133 件

**単発の事実・用語・手順**を期待した場合。

- **用語確認**: 「Statsig の発音は？」「要領の英訳は？」「クロス集計表の正式名称は？」
- **手順確認**: 「Mac でゴミ箱をサイドバーに固定するには？」「Figma のズーム方向を変えるには？」
- **仕様確認**: 「被保険者番号は何桁？」「OS ごとのファイル名上限は？」
- **事実確認**: 「法人と個人の資産はどちらが大きい？」「社保料が負になるケースはある？」

判定基準: AI の応答 1 回で用が足りるか。追加質問があっても、それは最初の質問の延長ではなく別の事実確認である場合が多い。平均クエリ数 1.5 回。

### 1.3 インパクトレベル（`impact`）

2 軸の組み合わせから機械的に導出する。

|  | 成果物 | 理解 | 事実 |
|:--|:--|:--|:--|
| **素材あり** | 共創 | 共創 | 検索 |
| **文脈あり** | 共創 | 探索 | 検索 |
| **質問だけ** | 試行 | 探索 | 検索 |

#### 共創（`co_creation`）: 85 件 / 29%

自分の素材や文脈を起点に、AI と一緒に成果物を作るか、設計を深める。

- 素材あり × 成果物（41 件）: ER 図→OpenAPI、職歴→英文履歴書、ワイヤーフレーム→HTML
- 素材あり × 理解（27 件）: API 設計の壁打ち、命名相談、仕様レビュー
- 文脈あり × 成果物（17 件）: Chrome 拡張開発、比較調査レポート

平均 3.1 クエリ。入力が充実しているため、AI の出力も構造化されやすい。

#### 探索（`exploration`）: 63 件 / 21%

対話で知識を掘り下げる。素材は不要だが、複数回のやりとりが発生する。

- 文脈あり × 理解（31 件）: 社保制度のエッジケース、DB 設計の相談
- 質問だけ × 理解（32 件）: PlantUML の使い方、GitHub 操作、素粒子物理

平均 3.2 クエリ。共創と同程度の対話の深さがある。入力が軽くても、対話の中で知識が積み上がる。

#### 検索（`lookup`）: 133 件 / 45%

事実を確認する。1-2 回で完結する。

- 質問だけ × 事実（96 件）: 発音、英訳、ツール操作、用語確認
- 文脈あり × 事実（30 件）: 背景つきの調べもの
- 素材あり × 事実（7 件）: コードを貼って「これは何？」

平均 1.5 クエリ。全体の 45% を占める最頻出パターン。AI を辞書・検索エンジンの代わりに使っている。

#### 試行（`trial`）: 14 件 / 5%

指示だけで成果物を作らせる。入力が軽いのに出力は重い。

- 顔写真生成（5 件）: うち複数が生成拒否で失敗
- コード生成（4 件）: HTML ラジオボタン、Stylus CSS、UUID 生成
- データ生成（3 件）: マイナンバーサンプル、bi-temporal サンプルデータ
- コマンド生成（1 件）: ffmpeg ワンライナー
- 英単語探索（1 件）: H,R,I,S を含む英単語の列挙

平均 2.1 クエリ。成功すれば共創と同等の価値があるが、入力が不十分なため成功率にばらつきがある。特に画像生成はポリシー制約による失敗が頻発。

### 1.4 スレッド衛生（`thread_contaminated`）

入力×出力の分類とは**独立した指標**として記録する。1 つのセッション（スレッド）に 3 つ以上のトピックが混在していれば `contaminated` とする。

| | セッション数 | トピック数 |
|:--|--:|--:|
| **きれい**（1-2 トピック/スレッド） | 104 (85%) | 118 (40%) |
| **汚染**（3 トピック以上/スレッド） | 19 (15%) | 177 (60%) |

**15% のスレッドに 60% のトピックが集中している。**

スレッド衛生を独立指標とした理由:

1. **入力の量とは無関係。** 「質問だけ」でも新しいスレッドで聞けば後から探せる。逆に、丁寧に素材を渡しても、56 往復スレッドの 37 番目に埋もれれば再利用できない
2. **出力の質とは無関係。** 汚染スレッド内のトピックの平均クエリ数（2.3）ときれいなスレッドの平均クエリ数（2.4）に有意差はない
3. **散逸リスクは「入出力の質」ではなく「整理の怠慢」から生じる。** これは分析軸ではなく運用の問題

汚染が最も深刻なセッション:

| スレッドタイトル | トピック数 | クエリ数 |
|:--|--:|--:|
| HTML ラジオボタンの修正と解説 | 27 | 57 |
| UI テキスト改善：保険料入力指示 | 23 | 46 |
| Google スプレッドシートで行番号を特定する | 20 | 59 |
| クリップボードの実装と仕組み | 18 | 41 |
| Mac ウィンドウクリック操作の改善 | 11 | 19 |

## 5. 失敗パターン

失敗として語るのは入出力の質の問題ではなく、**スレッド衛生の問題**。

- 15% のスレッドに知識の 60% が詰まっている
- 最悪のケース: 1 スレッドに 27 トピック、57 往復。タイトルは「HTML ラジオボタンの修正と解説」
- このスレッドの中に、マルチテナント DB 設計の重要な議論 10 往復分が埋もれていた

これは AI の使い方の問題ではなく、**整理の問題**。入力が充実していても、出力が価値あるものでも、スレッドが汚染されていれば後から探せない。

## 6. アクションアイテム

### ① 入力を充実させる

質問だけ（48%）から文脈あり・素材ありへシフトすれば、同じ質問でも AI の出力の質が上がる。特にコードや設計図を持っているときは、口で説明するのではなく貼り付ける。

### ② 重い出力を求める

事実の確認（45%）は悪くないが、それだけで終わらせない。「これは何？」の次に「ではこう設計するとどうなる？」と続ければ、検索が探索に変わる。

### ③ スレッドを分ける

話題が変わったら新しいスレッドを立てる。15% の汚染スレッドに 60% の知識が埋もれている現状は、明日から改善できる。
