## Slide 4: 成功① 対話で知識を深掘りする（45秒）

### ① 専門家の時間を奪わず、制度のエッジケースまで自分で掘る

例：社会保険の制度理解

```
「社会保険における区分とは？」
  → 「坑内員とは？」
    → 「英語の公式表現は？」
      → 「取得区分コードは？」

「健保と国保の違いは？」
  → 「国保組合に等級はあるか？」
    → 「厚年非加入で健保加入のケースは？」
```

- 概念理解 → 具体例 → エッジケース → プロダクト仕様への落とし込み
- 「ここに例外があるかもしれない」と気づく探索プロセスが仕様策定に直結する

**何をしてほしかったか:** 探索・深掘り — 制度の全体像から例外パターンまで、対話を通じて段階的に理解を深める

**プロンプトに含めた情報:** 素材は渡していない（問いあり）。ただし前の回答を踏まえて次の質問を組み立てている。最初は「社会保険における区分とは？」という広い問い。返答を読んで意図と違うとわかったら「それじゃない。短時間労働者かどうか、坑内員かどうかを反映する区分について聞いてる」と軌道修正。そこから「英語の公式ドキュメントでは何と呼ばれている？」「取得区分コードは？」と専門性を上げていった。

> スピーカーノート:
> 「1つ目。社会保険の制度理解です。人事労務SaaSを作るには、制度のエッジケースまで把握しないと正しい仕様が書けない。」
> 「従来はこれ、社労士や制度に詳しいメンバーに聞くか、厚労省の通達を自分で読むしかなかった。前者は相手の時間を奪う、後者は時間がかかる。」
> 「AIに階段型に質問していくと、概念→具体例→例外→仕様判断を、自席で、自分のペースで回せる。誰の時間も奪わずにできるようになった。」
> 「ポイントは、AIの最初の回答が的外れでも、"それじゃない"と軌道修正できること。対話だからこそ、自分が本当に知りたいことに辿り着ける。」

## Slide 5: 成功② プロトタイプで即決（35秒）

### ② その場でデモして、その場で意思決定

- HTML/CSS のプロトタイプを AI で生成
- ステークホルダーとの会議でデモ
- **その場で意思決定まで進んだ**

Before: 「あとでモック作ります」→ 次の会議で共有 → フィードバック → 修正 → ...

After: 「これを見てください」→ 議論 → 決定

**何をしてほしかったか:** 生成・変換 — テキストやワイヤーフレームから、ブラウザで動く HTML/CSS を一発で作らせる

**プロンプトに含めた情報:** 2パターンあった。1つ目は、画面の構成・レイアウト・各UIパーツの振る舞いをテキストで網羅的に書き下したもの（フィルター条件のタグUI、ページネーション、テーブルの展開機能まで指定）。2つ目は、手書きのワイヤーフレーム画像3枚を貼り付けて「基幹業務SaaSの画面を、画像に従ってHTMLで出力して」と指示したもの。いずれもAIが一発でサイドバー・タブ切替・テーブル表示まで含む動作するHTMLを返してきた。

> スピーカーノート:
> 「2つ目。AIでHTML/CSSのプロトタイプを作って、それをステークホルダーとの会議でそのままデモしました。結果、その場で意思決定まで進んだ。」
> 「"あとでモック作ります"がなくなった。提案→議論→決定のサイクルが、1回の会議で完結する。PdMとしてこれはかなり大きいです。」
> 「やり方は2通り試した。テキストでUI仕様を書き下すパターンと、手書きワイヤーフレームの画像を貼るパターン。どちらも動くHTMLが返ってくる。」

## Slide 6: 成功③ 素材を渡して変換させる（50秒）

### ③ エンジニアに「レビューして」と渡せる仕様書を自分で書く

**入力：** 事業所・社保・労保を網羅した 200 行超の Mermaid ER 図

```
erDiagram
    workplaces { int id PK; string name; ... }
    SOCIAL_INSURANCE_COVERED_WORKPLACES { ... }
    HEALTH_INSURANCE_SETTINGS { ... }
    HEALTH_INSURANCE_RATE_HISTORIES::BI-TEMPORAL { ... }
    PENSION_INSURANCE_SETTINGS { ... }
    ...
```

「これらの ER 図を前提に、GET の API を設計して OpenAPI Spec を書いて」

⬇️

**出力：** Bi-temporal 対応・リソース指向の完全な OpenAPI YAML

```yaml
paths:
  /workplaces:
    get: ...
  /workplaces/{workplaceId}/insurance-histories:
    get:
      parameters:
        - name: target_date      # Bi-temporal 対応
          in: query              # 基準日で時点データを取得
  /social-insurance-workplaces/{id}: ...
components:
  schemas:
    WorkplaceSummary: ...
    InsuranceHistory: ...
```

**何をしてほしかったか:** 生成・変換 — 自分が設計したER図を入力として、それと整合するAPI仕様書を丸ごと生成させる

**プロンプトに含めた情報:** Mermaid記法のER図そのもの。事業所マスタ、所在地、昇給月、賞与月、社会保険適用事業所（代表者・健保設定・健保料率履歴・厚年設定）、労働保険適用事業所（代表者・共通設定・労災設定・雇用設定）の全テーブル定義とリレーション。テーブル10個超、200行超。これに「これらのER図を前提に、データ取得GETのAPIを設計してOpenAPI Specを書いて」という一文を添えた。なお、このER図自体も別のセッションでAIと壁打ちしながら育てたもの（競合サービスの構造調査 → テーブル命名の相談 → ER図の段階的な拡張）。

> スピーカーノート:
> 「3つ目。自分が設計したER図、テーブル10個以上、Mermaidで200行超のものを丸ごと貼り付けて、"これを前提にAPIを設計してOpenAPI Specを書いて"と頼みました。」
> 「返ってきたのが、リソース指向のパス設計、Bi-temporalデータの基準日クエリ、全エンティティのスキーマ定義を含む完全なOpenAPI YAML。」
> 「これまでは設計意図をエンジニアに"口で説明"していた。でもこれがあれば、"このドラフトをレビューして"と渡せる。議論の出発点が変わります。」
> 「ちなみにこのER図自体も、別のセッションでAIと壁打ちしながら作ったもの。競合のデータ構造を調べるところから始めて、命名やテーブル分割を相談して、段階的に育てた。つまりAIの出力が次のAIの入力になる、連鎖が起きている。」

## Slide 9: 整理してみる — 2つの軸（40秒）

### この差はどこから来るのか？ 2つの軸で整理する

**軸1: 自分がどれだけ準備したか**

| レベル | 意味 |
|:--|:--|
| **素材あり** | コードや設計図など構造化した情報を渡した |
| **問いあり** | 明確な質問や依頼はあるが、素材は渡していない |
| **流れ任せ** | 前の話題の延長。独立した問いとして設計していない |

**軸2: AI にどんな役割を頼んだか**

| レベル | 意味 |
|:--|:--|
| **生成・変換** | 新しい成果物を作らせる |
| **探索・深掘り** | 対話を重ねて知識を掘り下げる |
| **レビュー** | 成果物を見せて確認・添削を頼む |
| **回答** | 事実や知識を単発で答えさせる |

> スピーカーノート:
> 「この上手くいく/いかないの差はどこから来るのか。整理してみました。2つの軸で考えます。」
> 「1つ目は、自分がどれだけ準備してからAIに渡したか。素材を渡したか、質問だけか、それとも流れでなんとなく聞いたか。」
> 「2つ目は、AIにどんな仕事を頼んだか。何かを作らせたか、一緒に考えたか、確認してもらっただけか、答えを聞いただけか。」

## Slide 10: 2軸のクロス → 3つのレベル（45秒）

### この2軸を交差させると、AI利用のインパクトが3段階に分かれる

|  | 生成・変換 | 探索・深掘り | レビュー | 回答 |
|:--|:--|:--|:--|:--|
| **素材あり** | 守備範囲の拡大 | 守備範囲の拡大 | 守備範囲の拡大 | 日常の加速 |
| **問いあり** | 守備範囲の拡大 | 守備範囲の拡大 | 日常の加速 | 日常の加速 |
| **流れ任せ** | 散逸リスク | 散逸リスク | 散逸リスク | 散逸リスク |

- **守備範囲の拡大:** 本来は別の専門家に頼む仕事を自分で完結できた
- **日常の加速:** 検索や辞書の代わり。小さいが積み重なる
- **散逸リスク:** 脈絡なく使い、得た知識が再利用できなくなる

> スピーカーノート:
> 「この2つの軸を表にすると、こうなります。AIに重い仕事——生成や深掘り——を任せると、PdMとしての守備範囲が広がる。単に答えを聞くだけなら、日常の効率化。そして"流れ任せ"で使うと、何を任せたかに関わらず知識が散逸する。」
> 「さっきの成功事例を見ると、①はここ、問いあり×探索。②はここ、問いあり×生成。③はここ、素材あり×生成。全部"守備範囲の拡大"のゾーンにいる。」
> 「失敗事例は左下——"流れ任せ"の行。何をしてもらったかに関わらず、準備なしの流れで使うと散逸リスクになる。」

## Slide 14: まとめ（40秒）

### AI は「不可能→可能」ではなく「やらない→やる」の閾値を動かす

社保制度のエッジケースを全部調べることも、ER 図から API 仕様を書くことも、123 セッションの全件分析も、理論上は手作業で可能。
でも実際にはやらなかった。AI がコストを下げたことで**「やってみよう」**に変わった。

### 明日から使える 3 つのこと

**① AI に「重い仕事」を任せる**

生成・探索・レビュー。聞くだけで終わらせない。

**② 素材を渡す**

自分が持っている構造化された情報を入力にする。出力の質は入力の質で決まる。

**③ 話題が変わったらスレッドを分ける**

守備範囲を広げた分の蓄積を、失わないために。

---

ありがとうございました

> スピーカーノート:
> 「まとめです。AIは、できなかったことができるようになる魔法ではない。でも、"面倒だからやらなかったこと"を"やってみよう"に変えてくれる。」
> 「明日から使える3つのこと。1つ目、AIに重い仕事を任せる。聞いて終わりにしない。2つ目、素材を渡す。入力の質が出力の質を決める。3つ目、話題が変わったらスレッドを分ける。広がった守備範囲で得た知識を、失わないために。」
> 「ありがとうございました。」
