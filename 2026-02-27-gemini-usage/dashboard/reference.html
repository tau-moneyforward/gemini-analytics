目的1-2：何をしてほしかったか

各トピックに対して「AI に何をしてほしかったか」を1つ割り当て。結末パターンとの相関が最も強い次元。

類型件数定義代表例傾向
一緒に考える 85件 (28.8%) AI と対話しながら段階的に考えを深めていく。初回回答をもとに追加質問・条件変更・深掘りを重ねる 「社保における区分とは？」→「坑内員とは？」→「英語の公式表現は？」→「取得区分コードは？」 一発承認+追加深掘りの割合が最も高い (85%)。概念理解→詳細→実装の流れで知識が深まる
見てもらう 29件 (9.8%) 自分が作ったもの（コード・仕様書・画像）を見せて、レビュー・添削・改善を依頼 HTML コードを貼り付けて「正しい？」、TS 型定義を提示して「ER 図を作って」 一発承認+追加深掘りが 79%。素材があることで AI の回答精度が上がる
作ってもらう 46件 (15.6%) 成果物（コード・文書・図・データ）の生成を直接依頼 「ダミー従業員 CSV を生成して」「PlantUML で ER 図を作って」「Jira Epic を書いて」 一発承認+追加深掘りが 78%。出力形式が明確なため結果が安定しやすい
知識を聞く 74件 (25.1%) 事実・知識・やり方を1つ質問して回答を得る。「〜とは？」「〜のやり方は？」 「Statsig はどう発音する？」「JSON 配列の順序は保証される？」 一発承認が大半を占める (80%)。効率的だが深い洞察は得にくい
ついでに聞く 61件 (20.7%) 前のトピックと無関係な新しい質問を同一スレッドに投入 Mac 設定 → Chrome PiP → Homebrew → zsh → GraphQL → Mermaid（1スレッドに6トピック以上が混在） トピック転換が 48% と突出して多い。他の4類型とは構成が大きく異なる
結末パターンとの関係: 「ついでに聞く」だけがトピック転換の割合が突出して高い (48%)。他の4つは一発承認+追加深掘りが 78-85% でほぼ横並び。クロス分析ヒートマップで詳細を確認できる。

目的1-3：プロンプトに含めた情報

質問時に AI にどのような情報を渡したか。1トピックが複数パターンに該当することがある（例: コード・資料を添付しつつ出力形式も指定）。

類型件数定義代表例傾向
コード・資料の添付 77件 (26.1%) コード、HTML、CSV、仕様書、画像などの具体的な素材を添付して質問 TypeScript 型定義 → ER 図、画像(等級表) → OCR、200行の Mermaid ERD → OpenAPI YAML 高品質な出力の最大の促進要因。構造化された入力が構造化された出力を生む
背景・目的の説明 89件 (30.2%) 「あるシステムは〜」「PdM として〜」など 1-2 文の前提を付与 「HR SaaS の DB を設計しているのですが…」「社保の手続きで確認したいのですが…」 一発承認+追加深掘りの割合が高い。AI が回答の焦点を絞りやすくなる
出力形式の指定 43件 (14.6%) 文字数、形式、対象読者、出力フォーマットなどを指定 「表形式で」「3パターンで」「箇条書きで」「マークダウンで出力して」 回答の実用性が上がる。特に成果物依頼との組み合わせで効果的
役割の指定 1件 (0.3%) 「あなたは〇〇の専門家です」型の役割指示 ほぼ未使用。大きな未開拓領域 データ不足で効果は不明。プロンプト工学では有効とされる
質問文のみ 142件 (48.1%) 質問文のみで、背景情報や出力形式の指定を一切添えない 「Statsig の発音は？」「クリップボードの仕組みは？」「PulumiとOpenTofuの違いは？」 一発承認+追加深掘りは 70% 超で大半だが、コード・資料の添付時と比較するとトピック転換がやや多い
結末パターンとの関係: コード・資料の添付時は一発承認+追加深掘りの割合が高い。質問文のみでも知識確認型では十分に機能する。因果ではなく相関の関係。

目的1-4：結末パターン

各トピックのインタラクションがどのように終わったかを分類。他の3軸と組み合わせてクロス集計することで、どの条件下でどの結末が出やすいかを確認できる。

パターン件数定義代表例傾向
一発承認 124件 (42.0%) AI の最初の回答をそのまま採用。追加質問なしに終了 「Statsig の発音は？」→ 回答を得て終了、単純な知識確認や短いコード生成 最も頻出。「知識を聞く」「質問文のみ」との組み合わせで多い
追加深掘り 87件 (29.5%) 最初の回答に続けて、関連する追加質問や深掘りに発展 社保の区分を聞く → 坑内員とは？ → 英語の公式表現は？ と展開 「一緒に考える」との組み合わせで顕著に多い
修正リファイン 24件 (8.1%) 回答に不足や誤りがあり、修正を依頼して改善させた 生成されたコードにバグ → 修正依頼 → 修正版で目的達成 「作ってもらう」で比較的多い。イテレーションの一形態
エラー指摘 4件 (1.4%) AI の回答に明確な事実誤認やエラーがあり、ユーザーが訂正した 計算結果の誤り → 指摘して再計算 → 正しい結果を取得 件数は少ないが、ユーザーが検証した証跡
反論・議論 2件 (0.7%) AI の見解に対してユーザーが異なる立場から反論し、議論が発生 設計方針の是非について AI と議論を展開 極めて稀。対話的な使い方の未開拓領域
トピック転換 51件 (17.3%) 別の話題に移行。元のトピックへの回帰なし Mac 設定の質問中 → Chrome PiP の話に → さらに Homebrew に、と話題が散逸 「ついでに聞く」と強く相関。同一スレッド内のトピック変更の結果
放棄 3件 (1.0%) やり取りを途中で完全に打ち切り 画像生成を複数回試行 → どれも不満 → スレッドを離脱 3件全てが画像生成（H）。AI の能力限界に起因
読み方: 結末パターンは他の3軸（話題ジャンル・何をしてほしかったか・プロンプトに含めた情報）との組み合わせで見ることで意味が出る。クロス分析ページのヒートマップで任意の2軸の関係を確認できる。
