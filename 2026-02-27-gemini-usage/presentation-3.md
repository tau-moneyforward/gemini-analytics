## Slide 4: 成功① 対話で知識を深掘りする（45秒）

### ① 専門家の時間を奪わず、制度のエッジケースまで自分で掘る

例：社会保険の制度理解

```
「社会保険における区分とは？」
  → 「坑内員とは？」
    → 「英語の公式表現は？」
      → 「取得区分コードは？」

「健保と国保の違いは？」
  → 「国保組合に等級はあるか？」
    → 「厚年非加入で健保加入のケースは？」
```

- 概念理解 → 具体例 → エッジケース → プロダクト仕様への落とし込み
- 「ここに例外があるかもしれない」と気づく探索プロセスが仕様策定に直結する

**与えた入力:** 質問だけ — 素材は渡していない。「社会保険における区分とは？」という一言から始まり、AIの回答を読んで「それじゃない。坑内員かどうかを反映する区分について聞いてる」と軌道修正しながら、段階的に専門性を上げていった

**期待する出力:** 理解 — 1回の回答で完結する事実確認ではなく、対話を重ねて制度のエッジケースまで理解を深めることが目的

> スピーカーノート:
> 「1つ目。社会保険の制度理解です。人事労務SaaSを作るには、制度のエッジケースまで把握しないと正しい仕様が書けない。」
> 「従来はこれ、社労士や制度に詳しいメンバーに聞くか、厚労省の通達を自分で読むしかなかった。前者は相手の時間を奪う、後者は時間がかかる。」
> 「AIに階段型に質問していくと、概念→具体例→例外→仕様判断を、自席で、自分のペースで回せる。誰の時間も奪わずにできるようになった。」
> 「ポイントは、AIの最初の回答が的外れでも、"それじゃない"と軌道修正できること。対話だからこそ、自分が本当に知りたいことに辿り着ける。」

## Slide 5: 成功② プロトタイプで即決（35秒）

### ② その場でデモして、その場で意思決定

- HTML/CSS のプロトタイプを AI で生成
- ステークホルダーとの会議でデモ
- **その場で意思決定まで進んだ**

Before: 「あとでモック作ります」→ 次の会議で共有 → フィードバック → 修正 → ...

After: 「これを見てください」→ 議論 → 決定

**与えた入力:** 素材あり — 2パターンあった。1つ目は画面構成・レイアウト・UIパーツの振る舞いを網羅的に書き下したテキスト仕様。2つ目は手書きワイヤーフレーム画像3枚を貼り付けて「基幹業務SaaSの画面をHTMLで出力して」と指示したもの。いずれもAIが一発でサイドバー・タブ切替・テーブル表示まで含む動作するHTMLを返してきた

**期待する出力:** 成果物 — ブラウザで動くHTMLをそのまま成果物として使う

> スピーカーノート:
> 「2つ目。AIでHTML/CSSのプロトタイプを作って、それをステークホルダーとの会議でそのままデモしました。結果、その場で意思決定まで進んだ。」
> 「"あとでモック作ります"がなくなった。提案→議論→決定のサイクルが、1回の会議で完結する。PdMとしてこれはかなり大きいです。」
> 「やり方は2通り試した。テキストでUI仕様を書き下すパターンと、手書きワイヤーフレームの画像を貼るパターン。どちらも動くHTMLが返ってくる。」

## Slide 6: 成功③ 素材を渡して変換させる（50秒）

### ③ エンジニアに「レビューして」と渡せる仕様書を自分で書く

**入力：** 事業所・社保・労保を網羅した 200 行超の Mermaid ER 図

```
erDiagram
    workplaces { int id PK; string name; ... }
    SOCIAL_INSURANCE_COVERED_WORKPLACES { ... }
    HEALTH_INSURANCE_SETTINGS { ... }
    HEALTH_INSURANCE_RATE_HISTORIES::BI-TEMPORAL { ... }
    PENSION_INSURANCE_SETTINGS { ... }
    ...
```

「これらの ER 図を前提に、GET の API を設計して OpenAPI Spec を書いて」

⬇️

**出力：** Bi-temporal 対応・リソース指向の完全な OpenAPI YAML

```yaml
paths:
  /workplaces:
    get: ...
  /workplaces/{workplaceId}/insurance-histories:
    get:
      parameters:
        - name: target_date      # Bi-temporal 対応
          in: query              # 基準日で時点データを取得
  /social-insurance-workplaces/{id}: ...
components:
  schemas:
    WorkplaceSummary: ...
    InsuranceHistory: ...
```

**与えた入力:** 素材あり — Mermaid記法のER図そのもの。テーブル10個超、200行超。なお、このER図自体も別のセッションでAIと壁打ちしながら育てたもの（競合の構造調査 → テーブル命名の相談 → ER図の段階的な拡張）

**期待する出力:** 成果物 — ER図と整合するAPI仕様書を丸ごと生成させる。AIの出力をそのままエンジニアへのレビュー依頼に使う

> スピーカーノート:
> 「3つ目。自分が設計したER図、テーブル10個以上、Mermaidで200行超のものを丸ごと貼り付けて、"これを前提にAPIを設計してOpenAPI Specを書いて"と頼みました。」
> 「返ってきたのが、リソース指向のパス設計、Bi-temporalデータの基準日クエリ、全エンティティのスキーマ定義を含む完全なOpenAPI YAML。」
> 「これまでは設計意図をエンジニアに"口で説明"していた。でもこれがあれば、"このドラフトをレビューして"と渡せる。議論の出発点が変わります。」
> 「ちなみにこのER図自体も、別のセッションでAIと壁打ちしながら作ったもの。AIの出力が次のAIの入力になる、連鎖が起きている。」

## Slide 9: 整理してみる — 2つの軸（40秒）

### この差はどこから来るのか？ 2つの軸で整理する

**軸1: 与えた入力** — AIにどれだけ材料を渡したか

| レベル | 意味 |
|:--|:--|
| **素材あり** | コード・設計図・画像など構造化データを渡した |
| **文脈あり** | 背景説明や形式指定はあるが、構造化された素材は渡していない |
| **質問だけ** | 質問文のみ。背景も素材もなし |

**軸2: 期待する出力** — AIからどんな成果を期待したか

| レベル | 意味 |
|:--|:--|
| **成果物** | そのまま使えるコード・仕様書・文書を作らせる |
| **理解** | 対話を重ねて知識を深める・設計を検討する |
| **事実** | 単発の事実・用語・手順を答えさせる |

> スピーカーノート:
> 「この上手くいく/いかないの差はどこから来るのか。整理してみました。2つの軸で考えます。」
> 「1つ目は、自分がどれだけ材料を渡したか。構造化されたデータを渡したか、言葉で背景を伝えたか、それとも質問文だけか。」
> 「2つ目は、AIからどんな成果を期待したか。使える成果物を作らせたか、一緒に考えたか、事実を聞いただけか。」

---

## Slide 10: 2軸のクロス → 4つのインパクト（45秒）

### この2軸を交差させると、AI利用のインパクトが4段階に分かれる

|  | 成果物 | 理解 | 事実 |
|:--|:--|:--|:--|
| **素材あり** | 共創 | 共創 | 検索 |
| **文脈あり** | 共創 | 探索 | 検索 |
| **質問だけ** | 試行 | 探索 | 検索 |

- **共創:** 自分の素材や文脈を起点に、AIと一緒に作るか設計を深める
- **探索:** 対話で知識を掘り下げる。入力が軽くても、やりとりの中で知識が積み上がる
- **検索:** 事実を確認する。1-2回で完結。AIを辞書のように使う
- **試行:** 入力なしで成果物を求める。成功すれば大きいが、成功率にばらつき

> スピーカーノート:
> 「この2つの軸を表にすると、こうなります。入力が重くて出力も重いのが"共創"。対話で理解を深めるのが"探索"。事実の確認が"検索"。そして入力が軽いのに成果物を求めるのが"試行"。」
> 「さっきの成功事例を当てはめると、①はここ——質問だけ×理解で"探索"。②と③はここ——素材あり×成果物で"共創"。上手くいった事例はすべて表の左上に寄っている。」

## Slide 12: 4段階に集約する + スレッド衛生（45秒）

### 半分が「共創 + 探索」、そしてスレッド衛生という別の問題がある

| インパクト | 件数 | 割合 | 平均クエリ |
|:--|--:|--:|--:|
| **共創** | 85 | 29% | 3.1 |
| **探索** | 63 | 21% | 3.2 |
| **検索** | 133 | 45% | 1.5 |
| **試行** | 14 | 5% | 2.1 |

- 共創と探索で **50%** — AI を辞書以上に使えている
- 検索は 45% — 価値がないわけではない。小さいが積み重なる
- 共創・探索には平均 **3回以上** のやりとりが必要

**スレッド衛生は入出力の質とは別の問題:**

- 85% のスレッドはきれい（1-2トピック）。**しかし、残り15%に60%のトピックが集中**
- さっきの失敗事例はここ。入力が充実していても、スレッドが汚染されていれば知識は再利用できない

> スピーカーノート:
> 「4段階に集約するとこうなります。共創29%と探索21%、合わせて半数がAIを辞書以上に使っている。検索45%は辞書的な使い方。試行5%は入力なしで成果物を求めた場面。」
> 「そしてここが大事なポイントですが、スレッド衛生は入出力の質とは独立した問題です。質問だけで聞いても、新しいスレッドで聞けば後から探せる。逆に、丁寧に素材を渡しても、57往復スレッドの37番目に埋もれたら再利用できない。」
> 「85%のスレッドはきれいなんですが、残り15%に全トピックの60%が集中していた。これは入力や出力の質の問題ではなく、整理の問題です。」

## Slide 14: まとめ（40秒）

### AI は「不可能→可能」ではなく「やらない→やる」の閾値を動かす

社保制度のエッジケースを全部調べることも、ER 図から API 仕様を書くことも、123 セッションの全件分析も、理論上は手作業で可能。
でも実際にはやらなかった。AI がコストを下げたことで**「やってみよう」**に変わった。

### 明日から使える 3 つのこと

**① 入力を充実させる**

質問だけ（48%）から素材あり・文脈ありへ。出力の質は入力の質で決まる。

**② 事実確認で終わらせない**

「これは何？」の次に「ではこう設計するとどうなる？」と続ければ、検索が探索に変わる。

**③ 話題が変わったらスレッドを分ける**

15%のスレッドに60%の知識が埋もれている。これは明日から改善できる。

---

ありがとうございました

> スピーカーノート:
> 「まとめです。AIは、できなかったことができるようになる魔法ではない。でも、"面倒だからやらなかったこと"を"やってみよう"に変えてくれる。」
> 「明日から使える3つのこと。1つ目、入力を充実させる。質問だけで終わらず、持っているコードや図を貼る。2つ目、事実確認で終わらせない。もう一歩踏み込めば、検索が探索に変わる。3つ目、話題が変わったらスレッドを分ける。15%のスレッドに60%の知識が埋もれている状況は、今日から変えられます。」
> 「ありがとうございました。」
